# Worktree Isolation Verification

## The Rule

**Every prompt to a worktree-dispatched agent must open with an isolation-verification block. If the agent is not in an isolated worktree, it must `HARD_FAIL` immediately. No fallback, no self-correction, no silent operation in main.**

## Why

Phase 3 dispatched 7 agents in parallel via `Agent({ isolation: "worktree" })`. Two of them (`a1d17a3f` textfield-slider, `a25ece03` usagehint-markdown) committed directly to `main`, bypassing their isolated worktree branches. Investigation traced the root cause to **the orchestrator's prompts**, not the runtime:

- Every prompt began with `for /Users/brahn/src/github.com/bdmorin/glimpse-a2ui` (the main repo absolute path).
- Agents read that as "project path" and prepended `cd /Users/brahn/src/github.com/bdmorin/glimpse-a2ui` to most/all of their Bash commands.
- That `cd` overrode the runtime-assigned CWD (which WAS the worktree path).
- Bash sessions are fresh per-invocation, so the override was per-command.
- Successful agents got lucky by running at least one command without an explicit `cd`, which ran in the runtime CWD = worktree, and committing from there.
- Failed agents kept `cd`-ing to main for every command including the commit, landing the work on main's branch.

The runtime's isolation is *contractual at setup* (it creates the worktree and sets CWD) but *advisory at runtime* (the agent can `cd` away). Without an explicit guard, prompt habits leak across the trust boundary.

## The Check

Open every worktree-agent prompt with this block, verbatim or near-verbatim:

```text
## FIRST ACTION — Isolation Verification (MANDATORY)

Before doing ANYTHING else, run this single Bash command and inspect the output:

  pwd && git rev-parse --git-dir && git rev-parse --show-toplevel

Required output shape:
  - `pwd` MUST end with `/.claude/worktrees/agent-<your-agent-id>` (or similar isolated path)
  - `git rev-parse --git-dir` MUST contain `/worktrees/` (it will look like
    `/Users/.../glimpse-a2ui/.git/worktrees/agent-<id>`)
  - `git rev-parse --show-toplevel` MUST equal `pwd`

If ANY of these conditions fail, you are NOT in an isolated worktree.

In that case: STOP IMMEDIATELY. Do not proceed. Do not `cd` anywhere to "fix"
it. Do not commit anything. Write only your auditlog fragment with
`outcome: HARD_FAIL` and a Notes line stating "isolation not provided by
runtime; orchestrator must dispatch with explicit worktree path." Then end
your turn.

## Working Inside Your Worktree

Your runtime-assigned CWD is your worktree. DO NOT prepend
`cd /Users/brahn/src/github.com/bdmorin/glimpse-a2ui` (or any absolute path
to the main repo) to your Bash commands. That overrides your CWD and lands
your work on main's branch instead of your isolated worktree branch.

Use relative paths from your worktree root. Use absolute paths only when
referring to files outside the worktree (which should be rare and
deliberate).
```

## Why HARD_FAIL Instead of Self-Correction

Earlier intuition: the agent could detect missing isolation and `git worktree add` itself one. Rejected for three reasons:

1. **Concurrency.** The runtime is dispatching multiple agents in parallel. An agent self-creating a worktree under contention reproduces the exact race that caused the original failure.
2. **Audit clarity.** A HARD_FAIL outcome surfaces the dispatch failure to the orchestrator immediately. A self-correcting agent hides the runtime's failure to provide isolation.
3. **Trust boundary.** Self-correction means the agent makes assumptions about its environment. HARD_FAIL keeps the agent purely reactive to the environment the orchestrator established.

When the orchestrator sees HARD_FAIL with this Notes signature, the response is to re-dispatch with explicit worktree-path setup (manual `git worktree add` first, then dispatch with the path baked into the prompt).

## Why Not Just Fix the Runtime?

The runtime is Anthropic's `Agent` tool. We can't change it. But the runtime is also doing its job — it creates the worktree and sets CWD. The bug was at the prompt layer, where my repeated `/Users/brahn/...` references trained agents to override that CWD. Fixing this at the prompt layer is the correct fix; the runtime isn't broken.

## Mode A vs. Mode B (terminology)

The investigation found two distinct failure modes:

- **Mode A — Stale-ref worktree.** The runtime creates a worktree but at a `main` ref cached from session start, not current HEAD. Agent sees pre-Phase-0 state. Self-correction via `git rebase onto main` works. Mitigated by existing prompt language ("rebase if not at current main").

- **Mode B — Worktree bypassed.** The runtime creates a worktree, but the agent's prompt-induced `cd` habit causes it to operate on main instead. Worktree branch stays empty; runtime auto-removes it after agent completion (which is why `git worktree list` shows no entry post-hoc). Mitigated by THIS rule.

Both modes are addressed: Mode A by the existing rebase-on-stale rule, Mode B by the new HARD_FAIL-if-not-in-worktree rule.

## Application

- **All future worktree-agent prompts** must include the FIRST ACTION block above.
- **Existing in-flight agents** (Phase 3 multiplechoice-checkbox at time of writing) cannot retroactively gain this check — wait for them to finish, then evaluate via post-hoc inspection (`git worktree list`, branch tip, transcript).
- **Recovery dispatches** (re-doing textfield-slider and usagehint with proper isolation) must use the new rule.
