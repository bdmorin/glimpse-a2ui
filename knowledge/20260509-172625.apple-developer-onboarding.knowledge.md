# Apple Developer Onboarding for a2glimpse

This is the path from "I have an unsigned `dist/a2glimpse.app`" to "any Mac
can launch it without a Gatekeeper scare-screen." If you only ever run
a2glimpse locally on your own machine, you do not need any of this. Skip to
the bottom for that case.

## What an Apple Developer account actually is

Apple operates two related things:

1. **Free Apple ID** — what you sign into iCloud / Mac App Store with.
   Lets you sign code locally with an *Apple Development* certificate.
   That cert only validates on machines whose "Provisioning Profile" lists
   them. Practically useless for distribution; fine for "does this build run
   on my own laptop without a Gatekeeper warning."

2. **Apple Developer Program ($99/year)** — gates two things:
   - The ability to issue a *Developer ID Application* certificate. That cert
     lets *anyone's* Mac launch your signed app, anywhere.
   - The ability to submit binaries for *notarization* — Apple's automated
     malware-scanning service. Without notarization, modern macOS shows a
     "...cannot be opened because Apple cannot check it for malicious
     software" dialog the first time a user double-clicks.

For a2glimpse distribution you want #2. Free dev account is not enough.

### Personal vs Business entity

Enrollment asks: "are you enrolling as an Individual or an Organization?"

- **Individual** — cert reads `Developer ID Application: Brian Morin (TEAMID)`.
  Easy enrollment (Apple ID + payment info). One human owner. No D-U-N-S
  number needed.
- **Organization** — cert reads `Developer ID Application: Provision IAM, LLC (TEAMID)`.
  Apple requires a D-U-N-S number (free, ~1 week turnaround at dnb.com),
  proof of legal entity, and a "you are authorized to sign on behalf of this
  org" attestation. More setup; signs as the company.

Recommendation for a2glimpse: **Individual**, unless you specifically want
the LLC name on the cert for liability/branding reasons. You can always
re-enroll later as an org and re-sign; users will see a new identity and
some Sparkle-style auto-updaters care about identity churn, but for a POC
appliance it's fine.

## Enrollment, step by step

1. Go to <https://developer.apple.com>.
2. Top-right → **Account** → sign in with your Apple ID.
3. If first-time, the page invites you to **Enroll**. Click it.
4. Pick Individual. Provide name, address, payment.
5. Wait for confirmation email (usually minutes for Individual; days for Org).
6. Return to <https://developer.apple.com/account>. You should now have a
   **Membership** card with a **Team ID** (10-character alphanumeric string,
   e.g. `ABC123DEFG`). Save this — you'll need it for notarytool.

## The certificate hierarchy (only the relevant bits)

Apple's developer portal lists ~10 cert types. Most are irrelevant. Here's
the map:

| Cert type | What it's for | Do you need it? |
|---|---|---|
| **Developer ID Application** | Sign apps for distribution OUTSIDE the Mac App Store | **YES** — this is what we want |
| Developer ID Installer | Sign `.pkg` installers | Maybe later if you ship a `.pkg` |
| Apple Development | Local-only builds for testing | No — useless for distribution |
| Mac App Distribution | Mac App Store submissions | No — we're not on MAS |
| Mac Installer Distribution | MAS `.pkg` submissions | No |

Ignore everything else.

## Generating the Developer ID Application certificate

Apple needs a Certificate Signing Request (CSR) generated on the machine
that will hold the private key. The private key never leaves your machine
(unless you export it intentionally). Lose it → you can revoke and
regenerate, but any binaries signed with the lost key keep working.

1. Open **Keychain Access** (`/System/Applications/Utilities/`).
2. Menu: **Keychain Access → Certificate Assistant → Request a Certificate
   From a Certificate Authority...**
3. Fill in:
   - User Email Address: your Apple ID email.
   - Common Name: anything human-readable (e.g. `Brian Morin Dev ID CSR`).
   - CA Email Address: leave blank.
   - Saved to disk: checked. (NOT "Emailed to the CA".)
4. Save the `.certSigningRequest` file somewhere temporary.
5. Go to <https://developer.apple.com/account/resources/certificates/list>.
6. Click **+** → choose **Developer ID Application** → Continue.
7. Upload your CSR. Continue.
8. Download the resulting `developerID_application.cer`.
9. Double-click the `.cer` to install it into your login keychain. The
   private key (already in the keychain from step 1-3) and the certificate
   pair into a single signing identity.

Verify:

```bash
security find-identity -v -p codesigning
```

You should see at least one line ending with
`"Developer ID Application: Your Name (TEAMID)"`. That string is your
signing identity — copy it verbatim for `codesign --sign`.

## Storing identifiers securely with fnox

You'll have three identifiers worth keeping in `fnox`:

```bash
fnox set APPLE_TEAM_ID            # 10-char team id from Membership page
fnox set APPLE_NOTARY_APPLE_ID    # email associated with your Apple ID
fnox set APPLE_NOTARY_PASSWORD    # app-specific password (see below)
```

Do not bake these into shell rc files or check them into the repo. Even
`APPLE_TEAM_ID` is mildly sensitive (it ties a binary to a real-world
identity).

## App-specific password (required by notarytool)

`xcrun notarytool` cannot use your real Apple ID password — Apple requires
an app-specific password for any non-interactive flow.

1. Go to <https://account.apple.com>.
2. Sign in → **Sign-In and Security** → **App-Specific Passwords** →
   Generate one.
3. Label it something like `a2glimpse-notarytool` so you can revoke later.
4. Copy the resulting `xxxx-xxxx-xxxx-xxxx` string into
   `fnox set APPLE_NOTARY_PASSWORD`.

This is *not* the same as your Apple ID password. Generating one does not
weaken your account; revoking it later does not affect your Apple ID.

## Signing the bundle

Once the cert exists in your keychain:

```bash
SIGN_ID="Developer ID Application: Brian Morin ($(fnox get APPLE_TEAM_ID))"
codesign \
    --sign "$SIGN_ID" \
    --options runtime \
    --timestamp \
    --deep \
    dist/a2glimpse.app
```

Flags worth knowing:

- `--options runtime` — enables the **hardened runtime**. *Required* for
  notarization. Without it, notarization fails with a generic "ineligible"
  error.
- `--timestamp` — embeds a trusted Apple timestamp. Without it, your
  signature expires when the cert expires (usually 5 years out). With it,
  the signature stays valid forever.
- `--deep` — recursively signs every Mach-O inside the bundle. For
  a2glimpse there's only one binary, so `--deep` is technically redundant —
  but it costs nothing and matters the moment you bundle a helper.

Verify locally:

```bash
codesign --verify --deep --strict --verbose=2 dist/a2glimpse.app
```

Expected output ends with `: valid on disk` and `: satisfies its Designated
Requirement`.

## Notarization

Apple's notarytool wants a `.zip` (or `.dmg`/`.pkg`) — not the raw `.app`.

```bash
ditto -c -k --keepParent dist/a2glimpse.app dist/a2glimpse.app.zip

xcrun notarytool submit dist/a2glimpse.app.zip \
    --apple-id "$(fnox get APPLE_NOTARY_APPLE_ID)" \
    --team-id "$(fnox get APPLE_TEAM_ID)" \
    --password "$(fnox get APPLE_NOTARY_PASSWORD)" \
    --wait
```

`--wait` makes the command block until Apple finishes scanning. Typically
1-5 minutes. Output ends with `status: Accepted` on success.

If it fails, get the log:

```bash
xcrun notarytool log <submission-id> \
    --apple-id ... --team-id ... --password ...
```

The log explains exactly which Mach-O failed which check.

## Stapling the notarization ticket

Notarization records the result on Apple's servers. Stapling embeds the
ticket into the `.app` so Gatekeeper can verify offline:

```bash
xcrun stapler staple dist/a2glimpse.app
```

Verify:

```bash
spctl -a -vv dist/a2glimpse.app
```

Expected: `accepted` and `source=Notarized Developer ID`.

## Common gotchas

- **Sign inside-out.** If you ever bundle helpers, sign children before the
  parent. `--deep` does this for you on the first pass; manual chains
  do not.
- **Hardened runtime entitlements.** a2glimpse uses WebKit + Cocoa with no
  exotic capabilities — no entitlements file is needed. If you ever add
  e.g. `com.apple.security.cs.allow-jit`, you need to embed an
  `entitlements.plist` and pass `--entitlements path.plist` to `codesign`.
- **Team ID mismatch.** notarytool will refuse if the `--team-id` doesn't
  match the cert that signed the bundle. They must be the same team.
- **App-specific password expires?** It doesn't expire on a timer, but you
  can revoke it from <https://account.apple.com>. Generate a new one,
  update fnox.
- **Sandboxing.** Not relevant here — sandboxing is only required for the
  Mac App Store. Developer ID distribution does not require it.

## Verification (the user's perspective)

After signing + notarizing + stapling, anyone who downloads
`dist/a2glimpse.app` can launch it (modulo the Finder-launch alert this
appliance shows on purpose) without seeing Gatekeeper warnings. Test on a
Mac that has never seen the bundle:

```bash
spctl -a -vv /path/to/downloaded/a2glimpse.app
codesign --verify --deep --strict --verbose=2 /path/to/downloaded/a2glimpse.app
```

Both should pass.

## When you do NOT need any of this

If a2glimpse only runs on machines you control — your laptop, your CI
runner, fellow engineers who can `npm install` — you can skip the entire
Apple Developer Program. Run `npm run build:app` and use the unsigned
bundle directly. The unsigned bundle works fine when spawned as a
subprocess (the MCP bridge case); only Finder/double-click triggers
Gatekeeper, and even there a `xattr -dr com.apple.quarantine` on the
downloaded bundle clears the quarantine flag.

## Cost summary

| Item | Cost | Frequency |
|---|---|---|
| Apple Developer Program | $99/year | Annual renewal |
