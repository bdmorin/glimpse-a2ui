# Shadow-DOM CSS Scoping Pathology

## TL;DR — The trap

Writing this in body's `<style>` tag in `src/a2glimpse-host.html`:

```css
a2ui-button[data-primary] { --a2g-btn-bg: var(--a2g-ink); }
a2ui-button:hover { ... }
a2ui-text code { font-family: var(--a2g-mono); }
```

**looks correct, parses fine, and silently does nothing.** The browser parses the rules, even reports `el.matches('a2ui-button[data-primary]')` as `true` for matching elements, but `getComputedStyle()` shows the declared values never apply.

**Cause:** every `a2ui-*` renderer component is a Lit element with its own shadow root. The surface tree lives **two shadow roots deep** from `<body>`:

```
<body>
  <a2glimpse-app>            ← Lit host, owns shadow root #1
    #shadow-root              ← contains <a2ui-surface>
      <a2ui-surface>          ← Lit host, owns shadow root #2
        #shadow-root          ← contains the entire surface tree
          <a2ui-root>
            <a2ui-column>
              <a2ui-row>
                <a2ui-button> ← target, lives in shadow #2
```

CSS selectors **do not cross shadow-root boundaries**. Body's `<style>` only sees children of `<body>` in the same scope: `<body>` itself, `<a2glimpse-app>` (the host element). It cannot match anything inside shadow root #1 or shadow root #2.

**Custom property inheritance DOES cross shadow boundaries.** This is what makes theming via tokens work: `--a2g-paper`, `--a2g-ink`, `--md-sys-color-*` declared on body inherit down through every shadow boundary into the surface tree. Component inline styles like `style="background: var(--a2g-btn-bg)"` (set by the IIFE's `additionalStyles` channel) resolve those vars at render time and correctly read the body's values.

This asymmetry — tokens propagate, selectors don't — is what makes the bug invisible: simple theming works, sophisticated theming silently doesn't.

## Reproducer

Spawn a window with two buttons (one `primary:true`), inject a diagnostic that walks the shadow ancestry of any rendered button, and probe `getComputedStyle().getPropertyValue('--a2g-btn-bg')`:

```js
const findButtons = (root, acc = []) => {
  (root.querySelectorAll ? root.querySelectorAll('*') : []).forEach(el => {
    if (el.tagName === 'A2UI-BUTTON') acc.push(el);
    if (el.shadowRoot) findButtons(el.shadowRoot, acc);
  });
  return acc;
};
findButtons(document).forEach(btn => {
  const cs = getComputedStyle(btn);
  const chain = [];
  let n = btn;
  while (n && chain.length < 15) {
    chain.push(
      n.nodeType === 11 ? '#shadow-of-' + (n.host?.tagName || '?')
      : n.nodeType === 9 ? '#document'
      : n.tagName
    );
    n = n.parentNode || n.host;
  }
  // Send via window.glimpse.send({error: {message: ...}}) for stdout capture
});
```

Output (verified 2026-05-09 against vendored 0ff4d09524c9 host bundle):

```
btn_yes (primary=true) → A2UI-BUTTON < A2UI-ROW < A2UI-COLUMN < A2UI-ROOT
  < #shadow-of-A2UI-SURFACE < A2UI-SURFACE
  < #shadow-of-A2GLIMPSE-APP < A2GLIMPSE-APP < BODY < HTML < #document
```

Two shadow boundaries between `<body>` and `<a2ui-button>`.

## What this breaks if you didn't know

Any of these patterns in body's `<style>` block do nothing:

- `a2ui-button:hover { ... }` — hover lift, transform, transition
- `a2ui-button[data-primary] { ... }` — primary/secondary differentiation via wrapperPatch attribute
- `a2ui-button:focus-within { outline: ... }` — keyboard focus ring
- `a2ui-textfield:focus-within { ... }` — input field focus state
- `a2ui-card { margin: 4px 0; }` — host-element margin (does not apply)
- `a2ui-text code, a2ui-text pre { ... }` — markdown-emitted code styling
- `::part(...)` — only works if the renderer exposes parts (it doesn't, in this build)

Diagnostic signal that you've hit this trap: `el.matches(selector)` returns `true`, `getComputedStyle(el).getPropertyValue('--your-prop')` is empty, and `outline-color: lime !important` (or any unmissable test) doesn't paint.

## What still works at body level

These channels DO reach into the surface tree because they ride custom-property inheritance, which crosses shadow boundaries:

- `body { --a2g-paper: #fbfcfd; }` — token declarations propagate.
- `body[data-color-scheme="dark"] { --a2g-paper: #1c1f24; }` — mode-aware token swap propagates.
- `body { font-family: ...; font-feature-settings: ... }` — typography inherits.
- `body { background: ...; color: ... }` — body's own paint, plus `color` inherits to text descendants.
- `additionalStyles.{Button,Card,TextField,...}` — these are inline styles set by the IIFE's render() inside the relevant shadow tree, so they apply directly. This is the renderer's primary theming hook and it works as documented.

## The fix pattern

For selector-targeted rules, inject a `<style>` element directly into the shadow root that contains your target. Use Lit's `firstUpdated` hook on the relevant component class.

For the surface tree (buttons, cards, text fields, etc.) — inject into **`a2ui-surface.shadowRoot`**:

```js
customElements.whenDefined('a2ui-surface').then((SurfaceCtor) => {
  const orig = SurfaceCtor.prototype.firstUpdated;
  SurfaceCtor.prototype.firstUpdated = function (changed) {
    if (orig) orig.call(this, changed);
    if (this.shadowRoot && !this.shadowRoot.querySelector('style[data-a2g-surface-shadow]')) {
      const style = document.createElement('style');
      style.setAttribute('data-a2g-surface-shadow', '');
      style.textContent = `
        a2ui-button[data-primary] { ... }
        a2ui-button:hover { ... }
        /* etc. */
      `;
      this.shadowRoot.appendChild(style);
    }
  };
});
```

For markdown-emitted content (`<code>`, `<pre>`, `<h*>` from the markdown directive) — those render inside **each `a2ui-text` instance's own shadow root**, so inject per-instance:

```js
customElements.whenDefined('a2ui-text').then((TextCtor) => {
  const orig = TextCtor.prototype.firstUpdated;
  TextCtor.prototype.firstUpdated = function (changed) {
    if (orig) orig.call(this, changed);
    if (this.shadowRoot && !this.shadowRoot.querySelector('style[data-a2g-text-shadow]')) {
      const style = document.createElement('style');
      style.setAttribute('data-a2g-text-shadow', '');
      style.textContent = `
        code { font-family: var(--a2g-mono); ... }
        pre { ... }
      `;
      this.shadowRoot.appendChild(style);
    }
  };
});
```

CSS variables referenced inside those injected stylesheets resolve against the host element's computed style, which inherits all the way back to body. So `var(--a2g-ink)` inside the surface-shadow `<style>` resolves correctly.

## Why this is the right fix (and not a workaround)

Three reasons:

1. **Trust boundary intact.** No edits to the vendored IIFE. Same wrapperPatch pattern as the existing Tabs / Modal / Button-primary patches. Mutation is host-side, post-define.
2. **Idempotent.** The `data-a2g-*-shadow` marker attribute lets the patch refuse to inject twice if the lifecycle re-fires.
3. **Lifecycle-correct.** Hooking `firstUpdated` (Lit's "shadow root is ready" signal) means the injection lands at the earliest possible moment in the component's lifetime, and applies to every instance the renderer ever creates.

Alternatives considered and rejected:

- **`adoptedStyleSheets`** — same scope problem; you have to adopt into each shadow root individually anyway, and `<style>` element insertion is universally compatible whereas `adoptedStyleSheets` requires constructable stylesheets.
- **Edit `Button.styles` / `Surface.styles` directly** — would mutate the IIFE's read-only contract.
- **Use `::part()`** — the renderer doesn't expose parts on most components in v0.8. Cross-cutting fix would require IIFE changes.
- **Inject into `a2glimpse-app.shadowRoot`** — tested and proven insufficient: the surface tree is one shadow root deeper. Rules there don't reach it.

## Lever vocabulary update — supersedes prior `vendored-renderer-pathologies` framing

The four levers documented in `20260509-154525.vendored-renderer-pathologies.knowledge.md` are still real, but **lever 3 ("Light-DOM host-element selectors") was incorrectly characterized**. That lever doesn't reach renderer-component descendants of `<body>` — it only reaches `<a2glimpse-app>` itself. The actual five-lever surface is:

1. **Wrapper-level CSS custom properties on body** (theming via inheritance) — works for tokens, not selectors.
2. **`additionalStyles` per-component** (set inside the IIFE's render) — works.
3. **`@font-face` declarations on body** — global, pierces shadow DOM via Document-level font set. Works.
4. **WrapperPatch via `customElements.whenDefined(...).then()`** — was used for Tabs / Modal / Button-primary attribute reflection. Now also used for shadow-style injection.
5. **Light-DOM CSS for non-renderer elements** (body, a2glimpse-app host) — works for the document-level chrome only.

The third lever in the old framing ("light-DOM host selectors for `a2ui-card { margin }` etc.") was always silently broken; nothing in the snapshots or visual goldens tested whether that particular margin landed because the renderer's own `:host { display: block }` masked it visually.

## Diagnostic kit — reusable for future "did my CSS land?" questions

Add this to the host HTML temporarily, gated on a `?diag=1` URL param if you want it permanent:

```js
setTimeout(() => {
  const out = [];
  const findEl = (root, tag, acc = []) => {
    (root.querySelectorAll ? root.querySelectorAll('*') : []).forEach(el => {
      if (el.tagName === tag) acc.push(el);
      if (el.shadowRoot) findEl(el.shadowRoot, tag, acc);
    });
    return acc;
  };
  findEl(document, 'A2UI-BUTTON').forEach(btn => {
    const cs = getComputedStyle(btn);
    out.push({
      id: btn.id,
      attr: btn.hasAttribute('data-primary'),
      btn_bg: cs.getPropertyValue('--a2g-btn-bg').trim(),
      ancestry: walkAncestry(btn)
    });
  });
  window.glimpse?.send?.({ error: { message: 'DIAG ' + JSON.stringify(out) } });
}, 2000);
```

`walkAncestry` is the chain walker shown earlier. Routes through the `error` stdout channel (the only one Swift forwards from webview to stdout).
