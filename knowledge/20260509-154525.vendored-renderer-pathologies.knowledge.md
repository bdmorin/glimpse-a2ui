# Vendored Renderer Pathologies

## TL;DR

The vendored Lit v0.8 renderer at `src/a2glimpse-host.html` is treated as read-only — but "read-only" doesn't mean "uneditable from outside." Three levers reach into shadow-DOM components without touching the IIFE:

1. **CSS custom properties** — pierce shadow-DOM boundaries via `var()` reference.
2. **`defaultTheme.additionalStyles`** — documented per-component theme hook the renderer applies inside the shadow tree.
3. **Prototype patching** — `customElements.whenDefined('a2ui-X').then(...)` lets you wrap or override component methods from a wrapper-side script after the IIFE registers them.

Phase 3 also surfaced two real bugs in the vendored renderer that are workable around using levers 1+3.

## When To Reach For Each Lever

| Symptom | Lever | Why |
|---|---|---|
| Component renders but invisible / transparent | CSS custom properties | The renderer's shadow CSS likely references unresolved `--md-sys-*` tokens. Define them at `:root` or `body` and they cascade in. |
| Component renders but visual chrome is bare / native-default | `additionalStyles` theme hook | The renderer applies `additionalStyles[ComponentName]` *inside* the shadow tree. Documented hook; safe; doesn't touch the IIFE. |
| Component throws / shadow tree empty / lit-html commit aborts | Prototype patching | Renderer-internal bug. Wrap the offending lifecycle method (e.g. `update`, `render`) from a wrapper-side `<script>` after `customElements.whenDefined(...)`. |
| Component's *fixture* doesn't capture the relevant state (e.g. closed trigger) | Fixture-side change | Not a renderer issue. Adjust `test/fixtures/<name>.jsonl` or add a sibling fixture. |
| Component requires a runtime asset (font, icon set, markdown engine) | Out of scope for polish | This is bundling/productization. Document and DEFER. |

## Lever 1: CSS Custom Properties

**Pierce shadow DOM via reference, not selector.**

The Lit renderer's component CSS uses `var(--md-sys-color-*)` and `var(--md-sys-elevation-*)` extensively — Material 3 design tokens. These are NOT defined anywhere in the IIFE. Without them:

- Backgrounds are transparent.
- Borders are invisible.
- Elevation shadows don't render.
- The component looks like it isn't there. (See Phase 3 multiplechoice-checkbox slice for the canonical example.)

Fix: define the tokens in the wrapper `<head>` style, on `body` (or `:root`):

```css
body {
  --md-sys-color-primary: #2563eb;
  --md-sys-color-on-surface: #1f2328;
  --md-sys-color-outline: #d0d7de;
  --md-sys-color-surface-container-low: #f6f8fa;
  --md-sys-elevation-level1: 0 1px 2px rgba(15, 23, 42, 0.06);
  /* … */
}
```

CSS custom properties are inherited and cross shadow-DOM boundaries when the shadow CSS does `var(--name)`. The host page never sees the inner usage; the renderer's shadow CSS resolves correctly.

The current canonical token set lives in `src/a2glimpse-host.html` in the `body { … }` block. Add new tokens there.

## Lever 2: `defaultTheme.additionalStyles`

**The documented per-component theme hook.**

Search for `additionalStyles` in the IIFE (around line 10797 in the current bundle). The renderer iterates this object and applies the rules INSIDE each component's shadow tree. This is the renderer authors' intended extension point. Examples already shipping:

```js
additionalStyles: {
  Button: { appearance: "none", borderRadius: "8px", padding: "10px 14px", … },
  Card: { border: "1px solid #d0d7de", borderRadius: "8px", padding: "16px", … },
  Text: { body: { margin: "0" }, h1: { margin: "0", fontSize: "24px" }, … },
  TextField: { boxSizing: "border-box", width: "100%", border: "1px solid #d0d7de", … },
  Slider: { appearance: "auto", width: "100%", accentColor: "#0a84ff", … }
}
```

Component name is the schema's PascalCase name, not the element selector (`Button`, not `a2ui-button`). For `Text`, the value is keyed by `usageHint` value (`h1`, `h2`, `body`, `caption`, …).

**Limitation:** this hook applies to a wrapper element inside the component's shadow tree. It does NOT reach into nested shadow children. For example, `additionalStyles.CheckBox` styles the `<section>` wrapper, not the inner native `<input>`. If the component's shadow CSS hard-codes `input { display: block; width: 100%; }`, your `additionalStyles` can't override it. That's a renderer-side concern.

## Lever 3: Prototype Patching

**Wrap or override Lit lifecycle methods from a wrapper-side script.**

Use sparingly. Only when the renderer has an actual bug that breaks rendering (not for cosmetic fixes — cosmetic goes in `additionalStyles` or CSS variables).

Pattern:

```html
<script id="a2glimpse-fix-X">
  customElements.whenDefined('a2ui-X').then((Ctor) => {
    const orig = Ctor.prototype.update;
    Ctor.prototype.update = function (changedProps) {
      // … pre-fix the bad input here, e.g. fill missing theme keys …
      return orig.call(this, changedProps);
    };
  });
</script>
```

Place after the closing `</script>` of the IIFE, before `</body>`. The `customElements.whenDefined(...)` ensures the IIFE has registered the component first; you wrap the prototype method without modifying the IIFE.

Trust boundary stays intact: this is page-side JS in the trusted host, not stdin-driven. No `eval`, no `unsafeHTML`, no agent input flows through it.

**Trade-off:** prototype patches are version-fragile. If the renderer bundle is updated, the patch may need re-evaluation. Document each patch with the renderer hash (or version, when known) so future-Brian knows what was fixed and where.

## Confirmed Renderer Bugs (workarounded in this repo)

### Bug 1 — `Tabs.render()` crashes on undefined `theme.components.Tabs.element`

**Symptom:** Tabs component renders blank. Shadow tree empty.

**Stack trace (via temporary diagnostic listener):** `keys@[native]` → `classMap.render` → `Tabs.render` → `Root.update`.

**Root cause:** `defaultTheme.components.Tabs` defines `{ container, controls: { all, selected } }` but **no `element` key**. `Tabs.render()` calls `classMap(this.theme.components.Tabs.element)`. lit-html's `classMap` does `Object.keys(undefined)` → throws → unhandled rejection mid-update → Lit aborts the shadow-DOM commit.

**Fix (Lever 3):** `customElements.whenDefined('a2ui-tabs').then((Ctor) => { … })` patches `Ctor.prototype.update` to ensure `theme.components.Tabs.element` exists before delegating to the original. See `src/a2glimpse-host.html` (the wrapper `<script id="a2glimpse-fix-tabs">` block).

**Discovered by:** Phase 3 tabs slice (agent `a56fd9a1aa9dea681`).

**Upstream status:** not filed yet. Worth filing against `google/A2UI` if/when we do an upstream pass.

### Bug 2 — MultipleChoice schema/element naming mismatch (`type` vs `variant`)

**Symptom:** A2UI v0.8 spec accepts `MultipleChoice.type` as a property. Lit element reads `.variant`. Fields the agent sets via spec are silently dropped.

**Root cause:** spec/element divergence inside the vendored renderer. The schema validator accepts `type`; the runtime resolver doesn't remap it to `variant`.

**Fix:** not yet shipped. Documented as DEFERRED in `knowledge/log/20260509-202432.phase3-multiplechoice-checkbox.auditlog.md`. A wrapper-side patch could remap incoming messages, but this is renderer logic and deserves an upstream fix.

**Discovered by:** Phase 3 multiplechoice-checkbox slice (agent `a6c45b3d6d5f09d4c`).

### Bug 3 — `MultipleChoice.selections.literalArray:[]` triggers exception

**Symptom:** `getCurrentSelections()` falls through to `processor.getData(comp, undefined, ...)` and throws.

**Root cause:** literalArray empty-array case isn't handled in the renderer's selection resolver.

**Workaround in fixture:** use `selections.path:"<some-path>"` form instead of `literalArray:[]`. The fixture in `test/fixtures/multiple-choice.jsonl` was updated accordingly.

**Discovered by:** Phase 3 multiplechoice-checkbox slice.

**Upstream status:** also a renderer-side fix candidate.

## What the IIFE-Read-Only Rule Forbids

For clarity, the rule explicitly forbids:

- Editing the IIFE's contents (any line inside the `(function() { ... })()` block in `src/a2glimpse-host.html`).
- Re-bundling the renderer from source (that's a different task — vendoring).
- Modifying renderer behavior in ways that change the protocol (A2UI v0.8 wire format must stay intact).

It does NOT forbid:

- Wrapper `<head>` CSS (custom properties, host-element rules).
- Wrapper `<script>` blocks that run *after* the IIFE (prototype patches, signal handlers).
- The `defaultTheme.additionalStyles` object that the IIFE itself reads.
- Fixtures and tests under `test/`.
