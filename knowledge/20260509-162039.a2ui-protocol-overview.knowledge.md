# A2UI Protocol Overview

## TL;DR

A2UI ("Agent-to-UI") is **Google's open protocol for agents to push declarative UI to clients without sending executable code**. Apache-2.0, repo at `github.com/google/A2UI`. Provider-neutral — any LLM that can emit spec-compliant JSON drives it. Current version: **v0.8 (Public Preview)** as of 2026-04. v0.9 is draft.

## Ownership & Status

- **Maintainer:** Google, with contributions from CopilotKit and OSS community.
- **License:** Apache 2.0.
- **Status (as of 2026-05-09):** "Early stage public preview" — the spec calls itself functional but evolving. Treat as stable enough to build against, not stable enough to vendor blindly without watching upstream.

## Core Idea

Instead of agents emitting either (a) prose, (b) raw HTML/JS, or (c) tool calls that execute code in the client, A2UI agents emit **declarative component descriptions** as JSON. The client maintains a *pre-approved catalog* of UI components and renders the agent's payload by mapping declared components to its trusted implementations. **No executable code crosses the trust boundary.**

This is the security thesis verbatim from the repo:

> A2UI is a declarative data format, not executable code. Your client application maintains a 'catalog' of trusted, pre-approved UI components.

## Message Flow

```
user prompt → agent → A2UI JSON stream → transport → client renderer
                                                     ↓
                                            renders via native catalog
                                                     ↓
                            user interaction → action message → agent
```

The wire format is a **flat list of components with ID references** (per the repo description). Designed for streaming/incremental generation by LLMs — the client can render partial trees as they arrive. Confidence: VERIFIED on the high-level shape; specific message-type names (e.g. `surfaceUpdate`, `dataModelUpdate`) appear in OpenClaw integration docs but were NOT enumerated in the A2UI README we fetched. Before relying on exact field names, **read `specification/` in the repo**.

## Reference Renderers

Confirmed in repo:
- **Lit (Web/TypeScript)** — used in their quickstart
- **Flutter** — via GenUI SDK
- **Angular**, **React** — listed as supported

Repo language breakdown: **TypeScript 78.5%**, Python 12.1%, Kotlin 3.8%, JavaScript 2.1%. The Lit web renderer is the most mature target for embedding in a WebView.

## Security Model — Specifics

1. **Declarative-only:** No `<script>`, no `eval`, no event handlers as strings. The renderer's component catalog defines what's renderable.
2. **Catalog-controlled:** Client decides which components exist. An agent can request `Button` only if the client implements `Button`.
3. **Action surface is bounded:** User interactions return as structured `action` messages with the component ID and payload — not arbitrary callbacks.
4. **Sandbox responsibility:** Spec language puts sandboxing on the client developer, not the protocol. ("security firmly in the developer's hands")

## Why This Matters for glimpse-a2ui

Upstream glimpse exposes `{type:"html", html:"<base64>"}` and `{type:"eval", js:"..."}`. Both ship executable code from agent to client. **Replacing those with A2UI message types means glimpse becomes safe to drive from any model/script without trusting the payload source.** That's the core thesis of the fork.

## Competing Hypothesis

A2UI could fail to gain traction and remain a Google-internal-flavored standard. Mitigation: the fork is useful even if A2UI stalls — a glimpse variant with a sane declarative payload format (vs. raw HTML/eval) is valuable on its own. Pin to v0.8 spec; if A2UI fades, the fork still works as a frozen vendored protocol.
