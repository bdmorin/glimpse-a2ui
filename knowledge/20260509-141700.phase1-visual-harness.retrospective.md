# Phase 1 Visual-Regression Harness Retro

## Verdict

The harness is **built and the capture pipeline is verified end-to-end**. Six fixtures render, snap-happy captures via mcporter, goldens write to a renderer-hash-keyed snapshot dir, the Swift `--test-mode` flag locks geometry, and `npm run test:visual:update` produces six valid PNGs that visibly contain the right components.

The acceptance criterion that failed is the comparison mode:

> `node test/visual.mjs` (no flag) passes byte-equal on a fresh run.

It doesn't. WKWebView captures are not byte-stable across runs on this host.

## What Works

- `--test-mode` + `A2GLIMPSE_TEST_MODE=1` — geometry locked (480x320 default, follow-cursor disabled, transparent/frameless/click-through forced off, `resize` and `follow-cursor` commands no-op while in test mode). Smoke test (`npm test`) still green.
- Six A2UI v0.8 fixtures under `test/fixtures/`: `button-only`, `card-text`, `modal`, `multiple-choice`, `text-field-form`, `tabs`. All valid v0.8 message shapes (`surfaceUpdate` + `dataModelUpdate` + `beginRendering`).
- `test/visual.mjs` orchestrates: spawn → ready → dispatch JSONL → 1.5s settle → ListWindows by title → TakeScreenshot → save. Routes through `mcporter call` exactly per the snap-happy-as-mcporter-shim convention.
- Renderer content-hash is baked into the snapshot path (`test/__snapshots__/<sha256-12>/<fixture>.png`). Bumping the renderer produces a new directory rather than silent drift.
- `npm run test:visual` and `test:visual:update` scripts.
- Manual inspection of goldens confirms 4/6 show the expected component visually (Button, Card+Text, Modal trigger, TextField+Button form). The other 2 (Tabs, MultipleChoice) render thinly — Tabs shows only window chrome, MultipleChoice shows the prompt label but not the options. Those are pre-existing renderer/theme issues identified in the POC retro and are precisely what Phase 3 component polish exists to fix. The harness's job is to lock down current state so polish work can detect regressions; Phase 1 is doing that.

## What Doesn't Work — The Real Finding

**Byte-equal comparison of WKWebView screenshots is not deterministic on this host.** Three back-to-back `--update` → compare cycles produced 2-5 spurious failures per run, with byte deltas typically 10-100 bytes (and occasionally hundreds of KB when capture briefly snagged the wrong window during heavy compositor activity).

Causes I'm confident about:

- **PNG encoder non-determinism.** Even pixel-identical raster data can produce different PNG bytes (filter heuristics, zlib variations).
- **Sub-pixel and AA jitter.** WKWebView fonts/borders rasterize with tiny variations frame-to-frame, especially around text edges.
- **Animation/blink state leak.** TextField shows a blinking caret when focused; capture phase varies. (`--test-mode` disables follow-cursor + window movement, but does not control renderer animations.)
- **Compositor focus-ring/shadow state.** macOS adds focus rings, shadow ramps that vary on capture.

Causes I am less confident about:

- One run produced a 370KB capture for `text-field-form` — visibly the iTerm window, not a2glimpse. Either snap-happy returned a stale windowId, or our window had been off-screen briefly. Did not reproduce on subsequent runs. I have a partial hypothesis (compositor delay between window creation and ListWindows visibility) but no proof.

## Why I Stopped Instead of Adding Pixel-Diff

Per `AGENTS.md` "Root Cause Over Workaround" and the plan's own `Don't paper over timeouts` rule by analogy: silently upgrading from byte-equal to pixel-diff would route around the failed assumption rather than surface it.

The plan said:

> Two modes: `--update` (write goldens) and default (compare; for v1, byte-equal is acceptable; pixel diff later).

That assumption is empirically false on this host. "Pixel diff later" needs to become "pixel diff now" — but that's a plan revision, not an implementation choice for me to make in flight.

Pixel-diff also opens design questions worth a real decision, not a sneak:

- **Tolerance threshold.** What's an acceptable per-pixel RGB delta? What's an acceptable percentage of changed pixels?
- **Dependency stance.** No-deps is a project value. ImageMagick `compare` is available on this host (`/opt/homebrew/bin/compare`); it could be shelled out to. Alternatively a tiny dev-only npm dep (`pngjs` + ~20 lines of diff code, or `pixelmatch`). Both are reasonable but the project has been zero-runtime-dep so far and this should be Brian's call.
- **Diff artifact format.** Save a side-by-side PNG? Save just the actual? Save a heatmap? `compare` can produce a visual diff image for free.

## Recommendation

Phase 1 is functionally done minus the comparison assertion. Recommend:

1. Decide tolerance + tooling stance (shell out to ImageMagick `compare`, or add a tiny dev dep).
2. Update `test/visual.mjs` to do tolerant pixel diff (e.g. fail if >0.1% of pixels differ by >threshold).
3. Re-bless goldens.
4. Update the plan doc to drop the "byte-equal is acceptable for v1" line.
5. Phase 2 hardening worktrees can dispatch immediately after that decision; the goldens and capture pipeline are ready.

## Risks Going Forward

- **Even with pixel-diff, captures can still vary by more than tolerance** if the renderer has true animations (TextField caret blink, focus ring). May need to add a short `disable-animations` shim — but that requires editing the host page, which the brief explicitly forbids ("Vendored renderer is read-only"). Workaround: capture a focus-neutral state (no field has focus). Need to verify.
- **Window-focus race.** The 370KB iTerm capture suggests there's a small window where ListWindows can return stale data or our window isn't yet visible. Adding a small additional wait + retry-once on size-anomaly would harden this. Not done — see Recommendation.
