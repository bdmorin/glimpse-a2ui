# Agent Dispatch Procedure

## Purpose

Codify the orchestrator-side procedure for dispatching worktree-isolated sub-agents on this repo. Read this before opening a session that will spawn agents. Following it isn't optional — a single skipped step in the parallel-dispatch case loses work or corrupts main (we've done both this session and survived; the procedure exists so we don't have to again).

## When To Dispatch (Decision Tree)

```
Is the task self-contained, well-scoped, and amenable to a 1-session brief?
  ├─ no  → do it yourself in the orchestrator session
  └─ yes → are there multiple INDEPENDENT slices?
              ├─ no, single task                        → 1 agent, sequential
              ├─ yes, but they touch the same file      → sequential agents OR
              │                                            single agent with bail-and-log
              └─ yes, fully independent                 → parallel agents
```

Single-agent dispatches are nearly always cheaper than they look — orchestrator saves context, agent gets focused brief, merge is trivial.

Parallel dispatches are tempting and useful, but each agent is ~1 session of orchestrator-prompted work plus merge cost. **Don't dispatch more than ~5 in parallel** unless you've staged section-marker scaffolding that isolates their edits to non-overlapping regions of shared files. Phase 3's 7 parallel touched the same `src/a2glimpse-host.html` style block; merge resolution was real work.

## Pre-Dispatch Checklist

Before calling `Agent(isolation: "worktree")`:

1. **`main` is clean and pushed (or at least committed).** Worktrees fork from `main`, but the runtime caches HEAD at session start — agents may land at a stale ref and have to rebase. Committing recent work increases the odds the runtime catches up; pushing makes the safety obvious.

2. **A plan or spec exists in `knowledge/`.** Agents need a self-contained brief that tells them: what the goal is, what's in scope, what's out, what the trust-boundary rules are, and what bail conditions look like. The brief lives in the prompt; the *deep* explanation lives in `knowledge/<timestamp>.<slice>.plan.md` (or the umbrella plan, like the Phase 1–4 polish-and-hardening plan).

3. **The relevant conventions docs are reachable from the prompt's required-reading list.** Specifically:
   - `knowledge/20260509-152436.worktree-isolation-verification.knowledge.md` (HARD_FAIL rule)
   - `knowledge/20260509-145843.parallel-agent-log-fragments.knowledge.md` (fragment-only logging)
   - `knowledge/20260509-154525.vendored-renderer-pathologies.knowledge.md` (lever vocabulary, when relevant)

4. **Branch / tag the current state if dispatch could touch many files.** A `git tag pre-<dispatch-name>-<date>` is cheap insurance against a recovery scenario. We did this once this session (`pre-recovery-2026-05-09`) and were glad to have it.

## Prompt Template

Every worktree-agent prompt follows this structure. Sections marked **MANDATORY** are non-negotiable.

```text
You are implementing **<Phase X — Slice Name>** for a2glimpse.

## FIRST ACTION — Isolation Verification (MANDATORY)

[Verbatim block from knowledge/20260509-152436.worktree-isolation-verification.knowledge.md]
- pwd && git rev-parse --git-dir && git rev-parse --show-toplevel
- HARD_FAIL conditions
- DO NOT prepend `cd /Users/brahn/src/...` to Bash commands
- Stale-ref handling: `git fetch . main && git rebase main`

## Required reading (in order)

1. AGENTS.md
2. The convention docs that apply (isolation, log fragments, renderer pathologies)
3. The plan or umbrella spec (e.g. polish-and-hardening-plan)
4. Any prior retros / fragments for this slice (especially if this is a retry)
5. The specific code files the agent will touch

## Goal

What the slice ships. Be concrete. List sub-tasks if there are several.

## Approach (suggested, not prescribed)

Steps that have worked for similar slices. Don't over-prescribe — give the agent room to make judgment calls.

## Bail-and-Log Rule

>1 session of fighting → DEFERRED fragment with rationale + estimated effort to revisit.
Specific bail triggers if known (e.g. "if Material Symbols requires bundling, defer").

## Constraints — durable

- Trust boundary intact (no html/file/eval, no new public stdin commands).
- Vendored renderer IIFE read-only. Wrapper is fair game.
- DON'T commit `test/__snapshots__/` (orchestrator owns cumulative re-bless).
- mcporter for snap-happy (never native MCP).
- Don't paper over timeouts (HARD_BLOCK with context, no silent retries).

## Reporting (MANDATORY format)

1. `knowledge/log/<timestamp>.<slice>.devlog.md` (NOT DEV_LOG.md directly).
2. `knowledge/log/<timestamp>.<slice>.auditlog.md` with `outcome:` field
   (OK | SOFT_BLOCK | HARD_BLOCK | HARD_FAIL | FAIL | DEFERRED | MIXED).
3. Retro at `knowledge/<timestamp>.<slice>.retrospective.md` if blocked.
4. Small themed commits in your branch. Do NOT merge.
5. Report back: (a) what changed, (b) acceptance results, (c) trust-boundary
   confirmation, (d) any deferrals with rationale.

Begin.
```

## Post-Dispatch Verification

**Within 10 seconds of `Agent` returning,** verify isolation took effect:

```bash
git worktree list | grep <agent-id>
```

If the worktree dir is missing, the runtime's `isolation: worktree` silently failed (Mode B from the Phase 3 investigation). The agent will run in the orchestrator's CWD and any commits land on the orchestrator's current branch. Options:

- Send an immediate redirect via `SendMessage` (the agent should HARD_FAIL on its first action anyway, given the FIRST ACTION block)
- Manually create the worktree (`git worktree add .claude/worktrees/agent-<id> -b worktree-agent-<id> main`) and re-dispatch with explicit path

## Mid-Session Monitoring

- **Do NOT poll** the agent's output file. The runtime sends `task-notification` events when agents complete; that's your signal.
- **Do NOT read** the JSONL transcript via the `Read` tool. They overflow context. If you must inspect (e.g. for debugging an isolation failure), use targeted `jq` filters or `grep` — see the Phase 3 investigation in `knowledge/DEV_LOG.md` for examples.
- **Stay productive while waiting.** Aggregate fragments from completed agents, write knowledge entries, do orchestrator-side maintenance — don't sit idle.

## Multi-Agent Concurrency Caveats

Observed during Phase 3:

- **Worktree creation race.** `git worktree add` calls under contention can silently fail. ~2 of 7 parallel dispatches in this session hit Mode B (no worktree). The HARD_FAIL convention catches it; agents stop instead of papering over.
- **mcporter / snap-happy concurrency.** Multiple agents running visual harness simultaneously cause snap-happy timeouts (window-id picking races). Workaround so far: agents retry-once on size anomaly; consider serializing visual-harness runs in future.
- **Cached-HEAD in worktrees.** Even successful isolations may branch from a stale `main` ref (`351775c` was cached at session start across the entire session). Agents must rebase as their second action. Don't skip the rebase instruction in the prompt.

## Merge Protocol (Orchestrator Owns This)

Sub-agents NEVER merge. The orchestrator pulls the work back via:

```bash
# 1. Fetch the agent's branch into a named local ref
git fetch . <agent-worktree-branch>:phase<N>-<slice>

# 2. Rebase onto current main (which may have moved since dispatch)
git checkout phase<N>-<slice>
git rebase main

# 3. Resolve conflicts manually. For DEV_LOG / AUDIT_LOG, prefer the
#    fragment convention — never edit those canonical files in agent
#    branches (they should write fragments only).

# 4. Fast-forward merge into main
git checkout main
git merge --ff-only phase<N>-<slice>

# 5. Verify
npm run build:macos && npm test && npm run test:visual

# 6. Delete the temp branch (worktree-bound branches stay)
git branch -d phase<N>-<slice>
```

**Cumulative re-bless of visual goldens** happens after ALL slices in a phase merge — not per slice. Each agent edits the host page, shifting the renderer hash; per-slice re-blesses would orphan goldens. After the phase's last merge:

```bash
npm run test:visual:update                          # capture at new hash
git rm -r test/__snapshots__/<old-hash>/            # drop stale dir
git add test/__snapshots__/<new-hash>/
git commit -m "test(visual): re-bless goldens at <new-hash>"
```

## Fragment Aggregation (At Phase End)

Per the log-fragment convention, agents write to `knowledge/log/<timestamp>.<slice>.{devlog,auditlog}.md`. Orchestrator concatenates into canonical logs at phase end:

```bash
cat >> knowledge/DEV_LOG.md <<HEADER

## YYYY-MM-DD — Phase N (per-slice fragments aggregated)

Fragments from parallel worktree agents, aggregated in dispatch order.
Original fragments preserved under knowledge/log/.

---

HEADER

for f in $(ls knowledge/log/*.devlog.md | sort); do
  slice=$(basename "$f" .devlog.md)
  ts=$(echo "$slice" | cut -d. -f1)
  name=$(echo "$slice" | cut -d. -f2-)
  printf '\n### %s — %s\n\nFragment: `%s`\n\n' "$ts" "$name" "$f" >> knowledge/DEV_LOG.md
  awk 'BEGIN{n=0} /^---$/{n++; next} n>=2{print}' "$f" >> knowledge/DEV_LOG.md
  printf '\n---\n' >> knowledge/DEV_LOG.md
done
```

Same pattern for `AUDIT_LOG.md` with `*.auditlog.md`. Fragments stay in `knowledge/log/` as durable per-slice retrospectives.

## Recovery Procedures

### If an agent commits to main (Mode B isolation failure)

1. Tag a safety net: `git tag pre-recovery-<date>`.
2. Identify the bad commits: `git log --oneline origin/main..main`.
3. Stash any orphaned untracked files: `git stash push --include-untracked -m "<dispatch-name>-orphans" -- <paths>`.
4. Branch the bad commits: `git branch <slice>-recovery <last-bad-commit-sha>`.
5. Rebase to drop unrelated commits if multiple agents bled together: `git rebase --onto <clean-base> <unrelated-commit> <slice>-recovery`.
6. Reset main to the clean base: `git reset --hard <clean-base-sha>`.
7. Re-merge each recovery branch in order, resolving conflicts manually.
8. Re-bless goldens cumulatively.
9. Document the incident in DEV_LOG.

The first time we did this, it took ~30 minutes. The second time it'll be 10. The HARD_FAIL convention should make a third time impossible.

### If multiple parallel agents cause merge conflicts in shared files

Expected. Phase 3 had 4 agents touching `src/a2glimpse-host.html`'s head style block. Resolution strategy:

- Merge in order of complexity (least-touching first, most-touching last).
- Keep the version with the richer styling / better comments / more thoughtful values.
- For `:root` token blocks specifically: don't keep multiple `:root { ... }` definitions — consolidate to one.
- Add cross-references in a single block: e.g. `a2ui-multiplechoice, a2ui-checkbox, a2ui-textfield, a2ui-slider { display: block; … }` instead of four separate rule blocks.

## Outcome Vocabulary (Standard)

- **OK** — work shipped, acceptance met, no caveats.
- **MIXED** — work shipped but with documented partial deferrals (e.g. button polished, button-icon deferred).
- **SOFT_BLOCK** — work in progress; encountered a decision point that requires orchestrator input. Agent reports its findings and stops.
- **HARD_BLOCK** — work cannot proceed without environment / runtime / infrastructure intervention. Includes timeouts, hung subprocesses, missing tools.
- **HARD_FAIL** — agent's pre-flight check failed (e.g. isolation verification). Agent did not begin actual work.
- **FAIL** — agent attempted work, hit unrecoverable error, work is invalid or partial-and-broken.
- **DEFERRED** — explicit decision to skip the slice's hard parts, with rationale and concrete next-iteration plan.

Auditlog fragment frontmatter must include `outcome:` with one of these values exactly.

## What This Session Cost

For calibration: this session ran ~16 worktree-agent dispatches across Phases 1–4, with:

- 2 Mode B isolation failures (textfield-slider, usagehint-markdown) → ~30 min recovery
- 1 Mode A stale-ref worktree on every agent → ~30 sec rebase per agent (not really a cost, just discipline)
- ~3 parallel-merge conflict sessions (Phase 2 logs, Phase 3 host CSS, log aggregations) → ~20 min total
- 1 mid-investigation `/superpowers:systematic-debugging` redirect from Brian → reframed isolation issue from "runtime bug" to "prompt-layer bug" in ~15 min

Net: agents shipped roughly 6× the orchestrator's direct-work throughput, with ~25% overhead for orchestration / recovery / merging. The procedure in this doc captures the post-hoc lessons that make the next session tighter.
