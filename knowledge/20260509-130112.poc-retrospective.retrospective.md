# a2glimpse POC Retrospective

## Verdict

The POC succeeded.

The important claim was not "A2UI looks polished out of the box." The important claim was: **Glimpse can be hardforked into an A2UI-native appliance without rebuilding the native shell from scratch.** That is now proven.

Working end-to-end:

- A native macOS `WKWebView` window starts from a single Swift binary.
- The host process sends A2UI v0.8 JSON Lines over stdin.
- Swift forwards A2UI messages into a trusted renderer host.
- The renderer builds a visible UI from A2UI components.
- User clicks produce structured `userAction` payloads on stdout.
- Invalid messages can report structured errors.
- A single-surface POC constraint is enforced.

This is enough plumbing to justify a next iteration if the goal is a coding-agent interaction surface.

## What We Actually Implemented

We implemented an A2UI transport appliance, not a full A2UI framework.

Implemented:

- JSONL process boundary: stdin A2UI messages, stdout lifecycle/actions/errors.
- Mac-native window lifecycle inherited from Glimpse.
- A trusted bundled WebView renderer host.
- A2UI v0.8 message dispatch for `surfaceUpdate`, `dataModelUpdate`, `beginRendering`, and `deleteSurface`.
- Single-surface rendering.
- Basic action bridge: Lit `a2uiaction` DOM events become A2UI-style `userAction` stdout messages.
- Basic literal/path context resolution for Button actions.
- A rough default theme sufficient for visual inspection.

Not implemented:

- Full multi-surface behavior.
- Catalog negotiation or inline catalogs.
- A2A envelope integration.
- A polished visual design system.
- A complete renderer compliance suite.
- Secure packaging/signing/notarization as a `.app`.
- Agent-side tool definitions, MCP server, or coding-agent hooks.

The actual A2UI interpretation is mostly delegated to Google's v0.8 Lit/Web Core renderer. Our bridge is the process, trust, packaging, and event boundary.

## Lessons From Porting Glimpse

The original architectural bet was accurate: Glimpse was already close to an A2UI appliance.

What transferred cleanly:

- JSONL framing maps well to A2UI streaming messages.
- `WKWebView` is a good rendering container for the reference Lit renderer.
- The existing native lifecycle, window flags, follow-cursor support, and stdin loop were reusable.
- The Swift single-file shape survived the fork.

What changed more than expected:

- The Node wrapper was deeply HTML/eval-shaped. Replacing it cleanly required a new `a2glimpse.mjs` rather than light edits.
- "Ready" has two meanings: WebKit navigation finished, and renderer host initialized. The POC currently treats WebKit ready as enough; a real version should wait for renderer-host ready.
- The public test harness needed a private synthetic click path. That is useful for CI but should not exist in production command handling unless explicitly gated.
- Trust boundary details matter immediately. Loading the renderer host from current working directory is convenient for development but too loose for a security-oriented appliance.

## Visual Debugging Findings

The visual issues are mostly **renderer host and theme integration issues**, not evidence that A2UI itself is wrong.

Observed:

- `Text` works, but `usageHint` headings render markdown markers literally because the host does not provide a markdown renderer.
- `Button`, `Card`, `Divider`, `Image`, `TextField`, `CheckBox`, `Slider`, `MultipleChoice`, `Tabs`, and `Modal` all render at least partially.
- `Icon` degrades without Material Symbols font availability.
- `Modal` placement and sizing are rough.
- Inputs function, but reference CSS/layout feels awkward in a bare host.
- The single-surface error renders clearly.

Likely fixes:

- Provide a markdown renderer or bypass markdown for heading usage hints.
- Bundle or map icons instead of assuming Google Material Symbols are available.
- Replace the minimal default theme with a purpose-built a2glimpse host theme.
- Add per-component visual fixtures and screenshots.
- Treat the Lit renderer as protocol machinery, while making the host/theme our product surface.

## Agent Integration Opportunity

The most promising next direction is a control surface for coding agents.

Good integration points:

- Coding-agent hooks can emit A2UI status surfaces for approval, progress, diffs, and choice prompts.
- User interactions can return through stdout as structured events, then be converted into hook responses or tool results.
- An MCP server can hold `a2glimpse` alive across tool calls and expose operations like `surface_update`, `data_model_update`, `close`, and `await_action`.
- A tool definition XML/schema can teach agents the allowed A2UI subset and reduce malformed output.
- Agent skills can package patterns: confirm dialog, multi-choice prompt, task progress, diff review, command approval, and form capture.

## Next Decisions

The next iteration should decide whether this remains a raw appliance or becomes a full agent UX layer.

Recommended next steps:

1. Harden the POC boundary: remove cwd renderer fallback, gate test-only commands, and wait for renderer-host ready.
2. Build a small "agent control surface" catalog of proven patterns rather than chasing every A2UI component.
3. Add an MCP/tool wrapper that owns the long-lived process and action correlation.
4. Write an agent skill that tells coding agents when and how to use a2glimpse.
5. Add visual fixtures/screenshots for the supported subset.

## Bottom Line

The plumbing is there. The remaining work is productization:

- polish the host renderer,
- define a supported A2UI subset,
- give agents a control surface,
- capture interactions through hooks/tools,
- and package the patterns so agents use it predictably.

This is no longer a "can it work?" question. It is now a "what should the first useful control surface be?" question.
