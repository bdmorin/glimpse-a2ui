# Parallel-Agent Log Fragments

## The Problem

Phase 2 dispatched three worktree agents in parallel. Each was instructed to append entries to `knowledge/DEV_LOG.md` and `knowledge/AUDIT_LOG.md` per the plan's conventions. Result at merge time: every parallel branch had appended at the same EOF region of both files, producing identical-shape merge conflicts on every merge after the first. Resolution was mechanical (keep both entries in dispatch order) but eats orchestrator time and risks losing content in a sloppy resolve.

This is structural. As long as agents directly edit shared append-only files, parallel dispatches will collide. Sequential dispatch sidesteps it but throws away the parallelism we want.

## The Convention

**Worktree agents write per-slice log fragments. Orchestrator aggregates at merge time.**

Concretely:

- Each agent writes its DEV_LOG-shaped entry to `knowledge/log/<YYYYMMDD-HHMMSS>.<slice>.devlog.md`.
- Each agent writes its AUDIT_LOG-shaped entry to `knowledge/log/<YYYYMMDD-HHMMSS>.<slice>.auditlog.md`.
- Agents **do not** edit `knowledge/DEV_LOG.md` or `knowledge/AUDIT_LOG.md` directly.
- At merge time, the orchestrator (a) merges the slice into main, then (b) concatenates the fragments into the canonical files in chronological order, (c) optionally deletes the fragments or (d) keeps them as durable per-slice artifacts.

The current call: **keep the fragments**. They're durable per-slice retrospectives, which is exactly what the agents already produce. Why throw them away after copying once?

## File Naming

```
knowledge/log/<YYYYMMDD-HHMMSS>.<slice>.devlog.md
knowledge/log/<YYYYMMDD-HHMMSS>.<slice>.auditlog.md
```

Where `<slice>` is the worktree's short identity (e.g. `phase2a-renderer-load`, `phase3-modal`). Filename sort = chronological order, which is the order canonical logs need.

The filename is intentionally redundant with frontmatter `type:` and content; the redundancy makes browsing-by-ls work without opening files.

## Fragment Frontmatter (required)

```yaml
---
type: devlog-fragment        # or: auditlog-fragment
date: 2026-05-09
slice: phase3-modal          # matches filename
agent: "Claude Opus 4.7"
worktree: worktree-agent-XXX
parent_session: 2026-05-09 orchestrator
outcome: OK                  # or SOFT_BLOCK / HARD_BLOCK / FAIL / DEFERRED — auditlog-fragment only
---
```

The body is the same shape as the original DEV_LOG / AUDIT_LOG entry (Author / Context / Did / Considered-rejected / Open-next for devlog; Agent / Worktree / Lineage / Action / Outcome / Artifacts / Notes for auditlog).

## Aggregation

Lightweight version (current): orchestrator concatenates manually after merge. One pass over `knowledge/log/<new-fragments>` per merge. Slot under correct timestamp in the canonical file.

Future version (when manual gets old): a small `scripts/regen-logs.mjs` that reads all fragments, concatenates in filename order, regenerates `DEV_LOG.md` and `AUDIT_LOG.md` wholesale. With this, the canonical files become *derived artifacts* — committed to the repo for readers, but never hand-edited.

Don't write the script before it's needed. Until parallel dispatch volume justifies it, manual aggregation is fine.

## Why Not Just Make the Canonical Files Indexes?

Considered: turn `DEV_LOG.md` into a one-line-per-entry index that links to fragment files. Rejected because:

- Index lines still get appended in parallel and still conflict — same problem one level shallower.
- A reader following the chronological narrative of the project benefits from full-text scrolling. An index forces N file opens.
- The fragment files exist regardless; concatenation is cheaper than indirection.

## Why Not Single Append-File With File Locking?

Considered: per-repo lock around DEV_LOG appends. Rejected because:

- Worktrees are isolated by design; cross-worktree locking defeats the parallelism.
- Doesn't compose with sub-agent dispatch (a future agent might dispatch its own children).
- Solves the wrong problem — the issue is shared mutable state across branches, not append ordering.

## Side Effect: Durable Per-Slice Retrospectives

Each fragment file is a complete, self-contained record of one agent's work on one slice: what it did, what it considered, what blocked it, what's open. Without this convention, retro information is buried in the canonical log's chronological soup. With it, you can answer "what did Phase 2c actually do, in detail?" by `cat knowledge/log/*.phase2c-*.{devlog,auditlog}.md` — no scrolling, no grep, no dependency on the canonical file's curation.

This is what Brian called out: "with DEV_LOG enhancements we get durable session retrospectives for free."

## Migration

Existing direct-append entries in `DEV_LOG.md` and `AUDIT_LOG.md` stay as-is. The convention applies forward to parallel-agent dispatches. Solo-orchestrator entries (this session, future single-thread sessions) may continue to append directly — the rule is "fragments when parallel," not "fragments always."

## Open Questions

- Should the convention extend to nested sub-agent dispatches (an agent that itself dispatches children)? Probably yes — a child agent writes its own fragment under `knowledge/log/`, which the parent or orchestrator aggregates. Not exercised yet.
- Should fragments retain forever, or roll into a `knowledge/log/archive/` after N days? Defer the decision until volume forces it.
