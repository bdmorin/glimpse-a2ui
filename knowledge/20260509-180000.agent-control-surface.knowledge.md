# Agent Control Surface

## Purpose

a2glimpse can render the full A2UI v0.8 component set, but coding agents do not need (and should not invent) the full set. The signal that matters is: **what kind of human-in-the-loop does the agent need right now?** This catalog answers that with seven canonical patterns. An agent picks the pattern that fits the moment, copies the JSONL skeleton, swaps in its own ids/labels/actions, and reads the returned `userAction`.

Pattern fit is the only decision. Component composition, theming, and layout details are the renderer's problem and the host's problem — not the agent's.

## Trust Boundary

This catalog is **guidance for the agent layer**, not a server-side validator. Nothing in `a2glimpse` reads this document. The MCP bridge (slice A2) is the validator: it MUST schema-check every incoming A2UI v0.8 message against the spec, reject `html` / `file` / `eval` shapes loudly, and only then forward to `a2glimpse`'s stdin. Agents may, in principle, send any v0.8-conformant payload through that bridge. This catalog is what they *should* send for routine human-in-the-loop work.

If you are building the MCP bridge: do not rely on agents using only these patterns. Validate the wire shape, not the pattern.

## The Seven Patterns

Pattern names are stable identifiers. Use them in fixture filenames, MCP tool argument hints, and skill documentation so agents and humans can pattern-match across the toolchain.

| name | criterion (one-liner) | reference fixture |
|---|---|---|
| `confirm` | yes/no decision; agent needs explicit human approval to proceed | (gap — proposed `test/fixtures/confirm.jsonl`) |
| `choice` | pick exactly one option from a small set (≤7) | (gap — proposed `test/fixtures/choice.jsonl`) |
| `multi-choice` | pick zero or more options from a set | `test/fixtures/multiple-choice.jsonl`, `test/fixtures/checkbox.jsonl` |
| `free-text` | capture a freeform string (or several) | `test/fixtures/text-field-form.jsonl` |
| `status` | read-only progress / state surface; no input expected | (gap — proposed `test/fixtures/status.jsonl`) |
| `diff-review` | present a code diff for accept/reject | (gap — proposed `test/fixtures/diff-review.jsonl`) |
| `command-approval` | present a shell command for approve/deny | (gap — proposed `test/fixtures/command-approval.jsonl`) |

Existing `button-only`, `card-text`, `modal`, `slider`, `tabs`, and `icon` fixtures are renderer-coverage probes, not agent patterns. They do not appear in this catalog. They stay where they are.

### 1. `confirm` — yes/no approval

**When to use:** the agent has decided what to do and needs explicit human go/no-go before acting. One question, two answers, irreversible-ish action on the other side.

**Don't use for:** "are you sure you want to quit?" type chrome — the agent shouldn't be asking that. Use for actions the agent itself is about to take.

**Example JSONL:**

```jsonl
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","component":{"Column":{"children":{"explicitList":["q","row"]}}}},{"id":"q","component":{"Text":{"text":{"literalString":"Apply 3 file changes to your repo?"}}}},{"id":"row","component":{"Row":{"children":{"explicitList":["no","yes"]}}}},{"id":"no_label","component":{"Text":{"text":{"literalString":"Cancel"}}}},{"id":"no","component":{"Button":{"child":"no_label","action":{"name":"confirm","context":[{"key":"answer","value":{"literalString":"no"}}]}}}},{"id":"yes_label","component":{"Text":{"text":{"literalString":"Apply"}}}},{"id":"yes","component":{"Button":{"child":"yes_label","primary":true,"action":{"name":"confirm","context":[{"key":"answer","value":{"literalString":"yes"}}]}}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[]}}
{"beginRendering":{"surfaceId":"s1","root":"root"}}
```

**Expected userAction:**

```jsonc
{"userAction":{"name":"confirm","surfaceId":"s1","sourceComponentId":"yes",
  "timestamp":"...","context":{"answer":"yes"}}}
```

`context.answer` is `"yes"` or `"no"`. Discriminate on that, not on `sourceComponentId`.

### 2. `choice` — single pick from a small set

**When to use:** the agent needs the human to pick exactly one option from 2–7. Above 7 options, prefer a TextField with autocomplete-style guidance or break into a tree of choices — long radio lists are a UX smell.

**Example JSONL:**

```jsonl
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","component":{"Column":{"children":{"explicitList":["q","pick"]}}}},{"id":"q","component":{"Text":{"text":{"literalString":"Which environment should I deploy to?"}}}},{"id":"pick","component":{"MultipleChoice":{"selections":{"path":"/env"},"maxAllowedSelections":1,"options":[{"label":{"literalString":"staging"},"value":"staging"},{"label":{"literalString":"production"},"value":"production"},{"label":{"literalString":"canary"},"value":"canary"}]}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[{"key":"env","valueString":"staging"}]}}
{"beginRendering":{"surfaceId":"s1","root":"root"}}
```

Pair with a submit `Button` if you want an explicit commit step; otherwise read the data-model path on the next event.

**Expected userAction:** comes back as a `MultipleChoice` selection event. Read `/env` from the next data-model snapshot the renderer emits, or wire a submit button with `context: [{key:"env", value:{path:"/env"}}]` to round-trip the value with the click.

**Renderer note:** A2UI v0.8 spec accepts `MultipleChoice.type:"radio"` for visual radio buttons; the vendored Lit element silently drops `.type` in favor of `.variant` (documented bug, see `vendored-renderer-pathologies`). For now, `maxAllowedSelections:1` is the load-bearing constraint; the visual treatment may render as checkboxes. Acceptable for the POC; revisit when the upstream bug is filed (HANDOFF.md B1).

---

### 3. `multi-choice` — zero or more from a set

**When to use:** "pick the files I should refactor", "which lints should I auto-fix", anything where the answer is a list, possibly empty.

**Example JSONL:** see `test/fixtures/multiple-choice.jsonl` for the canonical shape, or `test/fixtures/checkbox.jsonl` for individually-bound CheckBox components.

```jsonl
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","component":{"Column":{"children":{"explicitList":["q","pick"]}}}},{"id":"q","component":{"Text":{"text":{"literalString":"Which files should I refactor?"}}}},{"id":"pick","component":{"MultipleChoice":{"selections":{"path":"/files"},"type":"checkbox","options":[{"label":{"literalString":"src/auth.ts"},"value":"src/auth.ts"},{"label":{"literalString":"src/db.ts"},"value":"src/db.ts"},{"label":{"literalString":"src/util.ts"},"value":"src/util.ts"}]}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[{"key":"files","valueArray":[]}]}}
{"beginRendering":{"surfaceId":"s1","root":"root"}}
```

**Renderer gotcha:** do NOT seed `selections` with `literalArray:[]` — the renderer crashes in `getCurrentSelections()`. Always use the `path:"..."` form against an explicit data-model entry, as above. Documented in `vendored-renderer-pathologies` and HANDOFF.md B1.

**Expected userAction:** read `/files` from the data model. Pair with a submit Button to confirm.

---

### 4. `free-text` — capture freeform input

**When to use:** the agent needs a string the human types — commit message, search query, name, description. One TextField for one string; a Column of TextFields for a small form.

**Example JSONL:** see `test/fixtures/text-field-form.jsonl`. Skeleton:

```jsonl
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","component":{"Column":{"children":{"explicitList":["q","field","submit"]}}}},{"id":"q","component":{"Text":{"text":{"literalString":"Commit message:"}}}},{"id":"field","component":{"TextField":{"label":{"literalString":"Subject"},"text":{"path":"/subject"},"textFieldType":"shortText"}}},{"id":"submit_label","component":{"Text":{"text":{"literalString":"Commit"}}}},{"id":"submit","component":{"Button":{"child":"submit_label","primary":true,"action":{"name":"free-text","context":[{"key":"subject","value":{"path":"/subject"}}]}}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[{"key":"subject","valueString":""}]}}
{"beginRendering":{"surfaceId":"s1","root":"root"}}
```

Use `textFieldType:"longText"` for a multiline area.

**Expected userAction:**

```jsonc
{"userAction":{"name":"free-text","surfaceId":"s1","sourceComponentId":"submit",
  "timestamp":"...","context":{"subject":"<whatever the user typed>"}}}
```

### 5. `status` — read-only progress / state

**When to use:** the agent is in a long-running flow and wants to keep a window visible to the human while it works. No input expected; the agent updates the surface periodically with `dataModelUpdate` (and/or `surfaceUpdate` for structure changes).

**Don't use for:** spinners on transient operations. Use for multi-step tasks where the human will glance over while you work.

**Example JSONL:**

```jsonl
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","component":{"Card":{"child":"col"}}},{"id":"col","component":{"Column":{"children":{"explicitList":["title","step","detail"]}}}},{"id":"title","component":{"Text":{"usageHint":"h3","text":{"literalString":"Refactoring auth module"}}}},{"id":"step","component":{"Text":{"text":{"path":"/step"}}}},{"id":"detail","component":{"Text":{"text":{"path":"/detail"}}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[{"key":"step","valueString":"1 of 4"},{"key":"detail","valueString":"Scanning imports…"}]}}
{"beginRendering":{"surfaceId":"s1","root":"root"}}
```

To advance: emit additional `dataModelUpdate` lines updating `/step` and `/detail`. No close button needed — agent calls `{"type":"close"}` when done.

**Expected userAction:** none. This pattern is intentionally one-way.

---

### 6. `diff-review` — present a code diff for accept/reject

**When to use:** agent has a proposed code change and wants human approval before writing to disk. Differs from `confirm` in that the surface shows the diff body, not just the question.

**Example JSONL** (composition only — markdown rendering inside `Text` is out of scope per AGENTS.md, so the diff is shown plain):

```jsonl
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","component":{"Column":{"children":{"explicitList":["heading","path","diff_card","row"]}}}},{"id":"heading","component":{"Text":{"usageHint":"h3","text":{"literalString":"Apply this change?"}}}},{"id":"path","component":{"Text":{"text":{"literalString":"src/auth.ts"}}}},{"id":"diff_card","component":{"Card":{"child":"diff_text"}}},{"id":"diff_text","component":{"Text":{"text":{"literalString":"- function login(u, p) {\n+ async function login(user: string, password: string) {\n  // ...\n}"}}}},{"id":"row","component":{"Row":{"children":{"explicitList":["reject","accept"]}}}},{"id":"reject_label","component":{"Text":{"text":{"literalString":"Reject"}}}},{"id":"reject","component":{"Button":{"child":"reject_label","action":{"name":"diff-review","context":[{"key":"answer","value":{"literalString":"reject"}}]}}}},{"id":"accept_label","component":{"Text":{"text":{"literalString":"Accept"}}}},{"id":"accept","component":{"Button":{"child":"accept_label","primary":true,"action":{"name":"diff-review","context":[{"key":"answer","value":{"literalString":"accept"}}]}}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[]}}
{"beginRendering":{"surfaceId":"s1","root":"root"}}
```

**Expected userAction:**

```jsonc
{"userAction":{"name":"diff-review","surfaceId":"s1","sourceComponentId":"accept",
  "timestamp":"...","context":{"answer":"accept"}}}
```

**Limitation:** diffs render as plain text without syntax/diff coloring. Acceptable for POC. When markdown-in-Text or a dedicated CodeBlock component lands, revisit.

---

### 7. `command-approval` — present a shell command for approve/deny

**When to use:** agent wants to execute a shell command and the trust gradient says "ask first." Same shape as `diff-review` but the body is a single command and the action verbs are `approve` / `deny`.

**Example JSONL:**

```jsonl
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","component":{"Column":{"children":{"explicitList":["heading","cmd_card","reason","row"]}}}},{"id":"heading","component":{"Text":{"usageHint":"h3","text":{"literalString":"Run this command?"}}}},{"id":"cmd_card","component":{"Card":{"child":"cmd_text"}}},{"id":"cmd_text","component":{"Text":{"text":{"literalString":"$ rm -rf node_modules && npm install"}}}},{"id":"reason","component":{"Text":{"text":{"literalString":"Reason: dependency tree is corrupted; clean reinstall is the standard fix."}}}},{"id":"row","component":{"Row":{"children":{"explicitList":["deny","approve"]}}}},{"id":"deny_label","component":{"Text":{"text":{"literalString":"Deny"}}}},{"id":"deny","component":{"Button":{"child":"deny_label","action":{"name":"command-approval","context":[{"key":"answer","value":{"literalString":"deny"}}]}}}},{"id":"approve_label","component":{"Text":{"text":{"literalString":"Approve"}}}},{"id":"approve","component":{"Button":{"child":"approve_label","primary":true,"action":{"name":"command-approval","context":[{"key":"answer","value":{"literalString":"approve"}}]}}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[]}}
{"beginRendering":{"surfaceId":"s1","root":"root"}}
```

**Expected userAction:**

```jsonc
{"userAction":{"name":"command-approval","surfaceId":"s1","sourceComponentId":"approve",
  "timestamp":"...","context":{"answer":"approve"}}}
```

## Conventions Across Patterns

- **`action.name` mirrors the pattern name.** `confirm`, `choice`, `multi-choice`, `free-text`, `diff-review`, `command-approval`. Agents dispatch on `userAction.name` first, then `context`. (`status` has no action.)
- **Discriminate on `context`, not on `sourceComponentId`.** Component ids are scaffolding; `context.answer` and similar keys are the contract.
- **Submit-button pattern for forms.** Bind input components (`TextField`, `MultipleChoice`) to data-model paths. Carry the values out via `context: [{key:"x", value:{path:"/x"}}]` on the submit button. This avoids polling the renderer for intermediate state.
- **One pattern per surface.** Don't compose two patterns into one surface — patterns are designed standalone and rely on a single submit-button shape. To run two patterns concurrently (e.g. `status` + `diff-review`), use *two surfaces* and stack them. Multi-surface support shipped 2026-05-10 (HANDOFF.md C2): each `begin_rendering` adds to a vertical stack, each `delete_surface` removes; the window auto-grows to fit. Per-surface FIFO action queues let `await_action {surfaceId: "x"}` block on one surface while another keeps emitting state. See `skills/a2glimpse/SKILL.md` § Multi-surface for stacked examples.
- **`surfaceId` is opaque to the renderer.** Use a stable per-flow id so subsequent `dataModelUpdate` lines target the right surface.

## Forward References

- **A2 — MCP Bridge** will expose tools whose names mirror these patterns (e.g. `a2ui.confirm`, `a2ui.choice`, …) OR will expose the lower-level `a2ui.surface_update` / `a2ui.begin_rendering` / `a2ui.await_action` and let agents compose. The bridge owns trust-boundary validation. See HANDOFF.md A2.
- **A3 — Agent Skill** will teach coding agents *when* each pattern fits and *how* to phrase the MCP tool calls. The criterion column above is the seed for that skill's decision table. See HANDOFF.md A3.
- **B1 — Upstream filings** (`MultipleChoice.type` schema/element mismatch, empty-array selections crash) directly affect `choice` and `multi-choice`. Track there.

## Competing Hypothesis

The alternative shape we did not pick: expose every A2UI v0.8 component as a first-class agent primitive and let agents compose freely. Rejected because (a) coding-agent UX collapses to a half-dozen interaction shapes in practice, (b) free composition makes the MCP tool surface and the agent skill both sprawl, (c) a small named catalog gives us pattern-match leverage in fixtures, MCP tool names, and skill docs. If empirical use shows agents reaching for shapes outside the seven, expand the catalog — don't backfill the freedom we chose to constrain.
