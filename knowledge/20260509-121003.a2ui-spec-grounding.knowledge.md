# A2UI Spec Grounding for glimpse-a2ui

## Version Status

The local `google/A2UI` clone contains `specification/v0_8`, `specification/v0_9`, and `specification/v0_10`.

- `v0_8` is `0.8.2`, marked closed and no longer under active development.
- `v0_9` is the currently published closed specification.
- `v0_10` is currently under development.

This matters because the handoff pins the fork to v0.8, while the current Lit renderer README strongly recommends v0.9 for new projects.

## v0.8 Server-to-Client Message Types

The v0.8 spec confirms the provisional message names from the handoff:

- `surfaceUpdate`
- `dataModelUpdate`
- `beginRendering`
- `deleteSurface`

Each JSONL protocol message must contain exactly one of those top-level keys.

`surfaceUpdate` requires `surfaceId` and a non-empty `components` array.

`dataModelUpdate` requires `surfaceId` and `contents`; `path` is optional.

`beginRendering` requires `surfaceId` and `root`; `catalogId` and `styles` are optional.

`deleteSurface` requires `surfaceId`.

## v0.8 Client-to-Server Events

User interaction is represented as a client event wrapper containing `userAction` or `error`.

`userAction` includes:

- `name`
- `surfaceId`
- `sourceComponentId`
- `timestamp`
- optional `context`

The Lit v0.8 button implementation dispatches a DOM `a2uiaction` event with detail containing `action`, `dataContextPath`, `sourceComponentId`, and `sourceComponent`. The host page needs to listen for that event, resolve action context through the v0.8 processor, and forward the resulting `userAction` JSON to Swift via `window.glimpse.send(...)`.

## Lit Renderer Entry Point

There is no observed `window.a2ui.dispatch(...)` global in the v0.8 renderer source. The usable primitives are:

- `A2uiMessageProcessor` from `@a2ui/web_core/data/model-processor`
- signal-backed processor factory from `@a2ui/lit/0.8` / `@a2ui/lit/v0_8`
- `<a2ui-surface>` custom element from the Lit v0.8 UI package

The likely glimpse-a2ui embedding shape is a tiny host page/module that:

1. creates a signal-backed v0.8 `A2uiMessageProcessor`,
2. exposes a small local dispatch function that calls `processor.processMessages([message])`,
3. renders one `<a2ui-surface>` per visible processor surface,
4. listens for `a2uiaction` and emits v0.8 client events through the existing Glimpse bridge.

## Offline / Asset Notes

The v0.8 renderer source did not show obvious runtime `fetch(...)` or CDN dependencies in the inspected paths. The v0.9 explorer HTML does load Google Fonts, but that is demo-app code, not necessarily part of the renderer package.

This is not yet a proof that the renderer can run fully offline in WKWebView. The local A2UI clone has no `node_modules`, so the next step is to install/build the renderer and bundle a minimal v0.8 host page, then verify it under a custom URL scheme or file URL.

## Eval Harness

`google/A2UI/eval` is a model-generation evaluation framework built around Inspect AI, encrypted datasets, and provider-backed model runs. It is useful conceptually, but it is not a ready-made renderer compliance suite for this fork.

For the MVP, expect to hand-roll small JSONL fixtures and integration tests around:

- valid surface creation,
- Button rendering,
- click to `userAction`,
- invalid message rejection/error reporting,
- `deleteSurface`.

## Open Decisions for Planning

- Whether to honor the handoff's v0.8 pin or pivot the fork to v0.9 before the first implementation commit.
- Whether renderer assets should be vendored as source and bundled at build time, or checked in as a built static bundle.
- Whether the first UX should render multiple surfaces as stacked regions in one WebView or constrain MVP to one surface.
