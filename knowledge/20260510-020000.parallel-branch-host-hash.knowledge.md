# Parallel branches and the visual-golden host hash

## Why this exists

The visual harness in `test/visual.mjs` keys goldens by a content hash of `src/a2glimpse-host.html`. Goldens for hash `H` live at `test/__snapshots__/<H>/`. When the host file changes, the hash rotates and the harness reports "no golden for hash `<new-H>`" until you bless against the new hash.

This is **bundle-pin discipline**: tying goldens to renderer bytes prevents accidentally accepting a regression against a refactored host. The runbook for upstream-driven re-vendor is in [`20260509-172345.renderer-revendor-runbook.knowledge.md`](20260509-172345.renderer-revendor-runbook.knowledge.md).

What that runbook doesn't cover: **two branches that both edit `src/a2glimpse-host.html` at the same time.** That's the case this note addresses.

## What happens

When `feature-track` and `design-track` both modify the host file from the same `origin/main` baseline:

| Branch | Host hash | Goldens for it exist? | Harness says |
|---|---|---|---|
| `origin/main` | `H_main` (e.g. `0ff4d09524c9`) | Yes (blessed historically) | passing |
| `feature-track` | `H_feature` (e.g. `e8544dfcd584`) | No | "missing-golden" — looks like 9/9 fail |
| `design-track` | `H_design` (e.g. some other hash) | No | "missing-golden" — same shape |
| merged main | `H_merged` (a fourth hash) | No | "missing-golden" — until you bless once |

Critically: **the `H_main` directory continues to exist and represents the design-track baseline that design-track may still be working against.** Neither `feature-track` nor `design-track` should delete it.

## The wrong moves

These look reasonable in the moment and aren't:

- **`rm -rf test/__snapshots__/*/`** to "clean up old hashes." This nukes both `H_main` (design-track's baseline) and any in-flight bless from another branch. The glob expands to *every* hash directory. Don't.
- **Blessing your branch's full fixture set in isolation** (`npm run test:visual:update`). It produces 9 goldens for your hash that will be obsolete the moment your branch merges with the other one — the merge rotates the hash a final time. You've baked your work + only-your-half-of-the-host into a frozen baseline that nobody will use.
- **Comparing your hash's goldens against the other branch's hash** to "verify parity." The harness already keys by hash; goldens at `H_feature` and `H_main` aren't comparable except by eye.

## The right protocol

1. **On a feature branch, leave the goldens alone.** The harness will report "missing-golden" against your hash. Acknowledge it as expected; don't fix.
2. **For visual confidence during the slice, bless one fixture only.** `npm run test:visual:update -- --only <fixture>` produces a single PNG you can eyeball against the equivalent file under the prior hash directory. If the layout is what you intend, move on. If it's a regression, fix the host code, not the golden.
3. **Do not delete other hash directories.** They are read-only baselines for branches you can't see from inside your worktree.
4. **At merge, the merged branch is the only one that matters.**
   - Resolve conflicts in `src/a2glimpse-host.html` first (likely small — different sections of the file).
   - Run `npm run build && npm run test:visual` once. Confirm "missing-golden" for the merged hash.
   - Run `npm run test:visual:update`. This produces the full fixture set against the merged hash.
   - Visually diff the new goldens against the prior `origin/main` baseline (`H_main`). Flag anything unexpected; don't bless drift you can't explain.
   - Commit the new goldens in the same commit as the merge (or a single follow-up commit).
   - Optionally: prune obsolete hash directories. The branch hashes (`H_feature`, `H_design`) are now dead; only the merged hash matters going forward. The historical `H_main` can stay as a lineage breadcrumb or be pruned — preference is project-level.

## When this matters and when it doesn't

**Matters:**
- Both branches edit `src/a2glimpse-host.html`. (Even a one-line CSS change rotates the hash.)
- One branch is finished and the other is in flight, and the in-flight one wants to know whether their changes are visually clean.

**Doesn't matter:**
- Only one branch touches the host file. The other branch's goldens are untouched; standard single-branch flow applies.
- Both branches touch the file in *non-overlapping* ways but neither runs the visual harness until merge. The merge is the first hash rotation that requires a bless.

## Verified-by-merge: 2026-05-10 C2 + charcoal-workshop

The merge of `design-track` (charcoal-workshop theme + composite kitchen-sink fixture + per-fixture geometry override in the harness) into `session/feature-track` (C2 multi-surface) exercised this protocol end-to-end. What actually happened:

- **Hash trajectory:** `0ff4d09524c9` (pre-A3) → `7be8486d67c6` (mad-scientist-garage on main) → `a392c3eee7c5` (the merged hash). Each session held a working-only golden directory for its own intermediate hash; design-track had blessed `7be8486d67c6` as part of its own arc, feature-track did not bless intermediate state.
- **Merge mechanics:** `git merge --no-ff design-track` from `session/feature-track`. Auto-merge succeeded for `src/a2glimpse-host.html`, `src/a2glimpse.swift`, and `skills/a2glimpse/SKILL.md`. Conflict in `HANDOFF.md` only — both branches updated the milestone table independently. Resolution was a unified table covering both arcs chronologically.
- **Single bless:** `npm run test:visual:update` produced 10 goldens at the merged hash (9 pre-existing + the new kitchen-sink composite). 8 passed verify cleanly at the global 0.1% threshold; 2 (slider, kitchen-sink) failed with reproducible non-determinism in the native `<input type="range">` accent-color paint — see the per-fixture threshold note below.
- **Old hash directory removed in the same commit as the new one was added.** `git rm -r test/__snapshots__/<old>/` followed by `git add test/__snapshots__/<new>/` produces a clean diff that reads as "rebless," not "delete + recreate."

This means the merge-case is now VERIFIED, not just predicted. Confidence upgraded.

## Per-fixture threshold escape hatch

A practical refinement surfaced during the verified merge: fixtures containing native UA-rendered controls (in this case `<input type="range">` with `accent-color`) can exhibit reproducible sub-pixel non-determinism that the global 0.1% threshold doesn't tolerate. The harness now supports per-fixture override via `test/fixtures/<name>.meta.json`'s `pixelDiffThreshold` field; see `slider.meta.json` and `kitchen-sink.meta.json` for examples.

This is *not* a license to silence flakes. The threshold should be set narrow enough that real regressions still trip it (slider's 3% covers the accent-color flicker but a layout shift, color change, or geometry drift would exceed it). The `tolerance_note` field in the meta is a discipline anchor — if you raise a threshold, document why.

## A second non-obvious finding: harness contention

When all fixtures run sequentially via `npm run test:visual`, occasional WindowServer / mcporter-snap-happy contention causes fixture-specific custom-element shims (e.g. the test-mode `<a2ui-modal>` auto-open prototype patch) to miss firing for a single fixture. Re-running just that fixture (`--only modal`) succeeds at 0/153600.

This is not a goldens issue — it's harness fragility on shared system resources. Don't re-bless to "fix" a single sporadic full-suite failure; re-run first. If a single-fixture run also fails, then it's real. (This pairs with the existing E entry in HANDOFF.md flagging mcporter/snap-happy parallelism.)
