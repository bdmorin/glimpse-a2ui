# Investigation Methodology and Lever Vocabulary

## Why this doc exists

The session that landed `67774ae fix(theme): inject pseudo-class CSS into shadow scope where it can match` proved out two things bigger than the bug it fixed:

1. A **repeatable evidence-based technique** for "is this CSS rule actually applying?" questions, sized to fit in one diagnostic injection. We used it to crack a six-hour-class architectural bug in roughly thirty minutes once we stopped theorizing and started probing.
2. The **`wrapperPatch` pattern** — `customElements.whenDefined(<tag>).then(MutateLitClass)` — has now been used four times for four different concerns. It deserves an explicit name and an entry in the lever vocabulary, because every future "I want to change how the renderer behaves without editing the IIFE" question should default to it.

This document captures both, in a form that survives session compaction and gets indexed by global-knowledge agents alongside the bug-specific findings.

## Operating philosophy

Three principles, in priority order:

1. **Stop theorizing, start probing.** Every minute spent reasoning "could it be specificity? could it be cascade order? could it be inheritance?" without evidence is a minute the diagnostic injection would have answered. We theorized about var() resolution failure, focus-state lingering, primary-attribute reflection, and three other hypotheses before we just *probed* the matched cascade. The probe finished the case in seconds.

2. **Evidence has shape, ranked by directness.** Strongest to weakest:
    - DOM truth (`getComputedStyle`, `el.matches`, parentNode-walk) — observed reality.
    - Renderer JSON output / wire shape (what the IIFE thinks it received).
    - Visual output (screenshot) — but visual is the *consequence* of evidence, not evidence itself. Misread it and you chase ghosts. We did this twice in this session.
    - Reasoning from first principles — useful for ranking hypotheses, weak for confirming them.

3. **Trust the channels, name the boundaries.** The IIFE is read-only. The renderer's component classes are read-only. Body's CSS, custom properties, attributes, and Lit lifecycle hooks are mutable. Every fix should make the boundary between "stuff we own" and "stuff we don't" loud — comments, named constants, marker attributes (`data-a2g-*`). A future agent reading the codebase should not have to guess where the trust boundary is.

## The Three-Round CSS Cause-Isolation Technique

When a selector "should match" but the visual doesn't agree, run these three rounds in order. Stop as soon as evidence answers the question.

### Round 1 — Selector match check

Before anything else, prove the selector is parseable and JS-evaluable:

```js
btn.matches('a2ui-button[data-primary]');  // expect true
```

If this is `false`, your selector has a typo, the attribute isn't actually set, or the element isn't what you think it is. **STOP — fix the misconception.**

If this is `true` but the rule isn't applying, you have a real cascade or scoping issue. Continue.

### Round 2 — Specificity / `var()` resolution probe

Replace your declaration with the most direct possible value, plus `!important`:

```css
a2ui-button[data-primary] {
  --a2g-btn-bg: red !important;
  outline: 3px solid lime !important;
}
```

If `getComputedStyle(btn).getPropertyValue('--a2g-btn-bg')` is now `red` (or `outline-color` is `lime`), the original problem was `var()` resolution or a specificity loss. Find the conflicting rule.

If it's STILL not red, the rule itself isn't being applied. Specificity / cascade isn't the issue. Continue.

### Round 3 — Probe-property test

Add a custom property whose name you know nothing else sets, with a literal text value:

```css
a2ui-button[data-primary] { --probe-rule: PRIMARY-RULE-FIRED; }
a2ui-button { --probe-base: BASE-RULE-FIRED; }
```

```js
getComputedStyle(btn).getPropertyValue('--probe-rule').trim();
```

If `--probe-rule` is empty for a button that `matches()` returns `true` on, the CSS engine isn't matching the rule despite JS's `matches()` saying it should. **You're outside the scope.** This is almost always a shadow-DOM boundary problem.

When you hit Round 3 with empty probes, walk the ancestry chain:

```js
let n = btn, chain = [];
while (n && chain.length < 15) {
  chain.push(
    n.nodeType === 11 /*ShadowRoot*/ ? '#shadow-of-' + (n.host?.tagName || '?')
    : n.nodeType === 9 /*Document*/  ? '#document'
    : n.tagName
  );
  n = n.parentNode || n.host;
}
console.log(chain.join(' < '));
```

Any `#shadow-of-*` between your element and `body` means body's `<style>` cannot reach it via selector. You're now in the shadow-scoping problem documented at `20260509-214700.shadow-dom-css-scoping-pathology.knowledge.md`.

### Why three rounds, in this order

Each round eliminates a class of hypotheses:

- Round 1 eliminates "you typed the wrong selector" / "the attribute isn't there."
- Round 2 eliminates "specificity / cascade order / failed var() lookup."
- Round 3 eliminates "the rule is being evaluated but not applied."

The remaining cause space after Round 3 is roughly: shadow-scoping issue, animation/transition stuck on a different layer, or browser bug. Shadow-scoping is the most common in this codebase by far.

This is faster than the alternative (binary-search-by-edit), and it produces an audit trail you can paste into a knowledge doc.

## The `wrapperPatch` lever — name and pattern

Every fix in this codebase that mutates renderer behavior without editing the IIFE follows the same pattern:

```js
customElements.whenDefined('<tag>').then((Ctor) => {
  const orig = Ctor.prototype.<lifecycleHook>;
  Ctor.prototype.<lifecycleHook> = function (changedProps) {
    /* mutation here — runs on every instance, on every <hook> firing */
    if (orig) orig.call(this, changedProps);
    /* OR mutation after — sometimes order matters */
  };
});
```

`<lifecycleHook>` is one of:

- **`update`** — fires on every property change. Use when the mutation depends on a Lit `@property` value (e.g. reflecting `primary` to `data-primary`).
- **`firstUpdated`** — fires once after the shadow root is built. Use for one-shot setup like injecting `<style>` into shadow.
- **`updated`** — fires after every render, after DOM is in sync. Use for click-trigger or DOM-dependent post-processing (e.g. Modal auto-open).
- **`willUpdate`** — fires before update, can mutate state. Rarely needed.

### Established uses (as of 2026-05-09)

Four wrapperPatches live in `src/a2glimpse-host.html`. Each solves a different category of problem:

| Patch | Hook | Concern | Why it's a wrapperPatch |
|---|---|---|---|
| **Tabs theme-gap fill** | `update` | Renderer crashes on missing `theme.components.Tabs.element` | Renderer expectation gap — fill in the missing structure before render() reads it |
| **Modal auto-open in test mode** | `updated` | Visual-regression harness needs the open state | DOM-dependent click trigger; fires after render commits |
| **Button `primary` → `data-primary` reflection** | `update` | Lit `@property` doesn't auto-reflect; light-DOM CSS needs the attribute to differentiate primary/secondary | Attribute reflection that the original component declined to do |
| **Surface shadow-style injection** | `firstUpdated` | Light-DOM selector-based CSS rules can't reach into renderer shadow tree | Add a `<style>` to a shadow root that the IIFE owns |

Each one is small (5–15 lines), idempotent (safe to re-fire), and self-documenting (a comment + a `data-*` marker attribute when state needs to be visible).

### When to reach for `wrapperPatch`

Ask: "Do I need to mutate behavior of a renderer component, OR add behavior the renderer didn't include?"

- "Mutate" → patch the relevant lifecycle hook. Examples: missing default theme, missing attribute reflection, missing post-render side effect.
- "Add behavior" → typically inject something (event listener, child element, style) and trigger via the same hook. Examples: shadow-style injection, auto-clicking modal triggers, ARIA additions.

If you're tempted to fork the IIFE: stop. The IIFE is part of the trust contract. WrapperPatch first; fork only if the patch is fundamentally impossible (e.g. an internal private-field WeakMap with no public accessor and no event hook).

### When NOT to reach for `wrapperPatch`

- **Pure styling that custom properties can express.** If the change is "make Buttons amber on hover," route through `--a2g-btn-bg` from the surface-shadow `<style>` block instead. WrapperPatch is for behavior, not aesthetics.
- **One-shot orchestration logic** that lives in the orchestrator session (Swift host, MCP bridge). WrapperPatch is for "every instance, forever." Single-window setup belongs in Swift / TS bridge code.
- **Anything that would break the trust boundary.** No `eval`, no `innerHTML` from agent input, no surfacing of new public stdin commands without a corresponding bridge validator update.

## Lever Vocabulary v2 — the FIVE host-side levers (corrected)

The prior framing in `20260509-154525.vendored-renderer-pathologies.knowledge.md` listed four levers and characterized one of them ("light-DOM host-element selectors") as functional. **It wasn't.** It only worked for elements directly in body's light DOM (i.e. `<a2glimpse-app>` itself, and rules like `body { background }`). The renderer-component selectors it was supposed to enable (`a2ui-card { margin }`) silently did nothing, masked by the renderer's own `:host { display: block }` defaults.

Corrected vocabulary:

### Lever 1 — CSS custom properties on body (theming via inheritance)

What it is: `body { --a2g-paper: #fbfcfd; --md-sys-color-primary: ...; }` etc.

What it reaches: every descendant of body, **including across shadow-DOM boundaries**, because custom property inheritance crosses shadow scopes by spec.

What it can do: token swaps for theming (light/dark mode, accent colors, mode-aware values). Anything where "the renderer reads `var(--name)` somewhere" is the trigger.

What it CAN'T do: state-driven styling (`:hover`, `[data-*]`), descendant selectors, anything requiring the CSS engine to *match* into the shadow tree.

Examples in this codebase: every `--md-sys-color-*`, every `--a2g-*` token. Dark-mode swap on `body[data-color-scheme="dark"]`. The font stack.

### Lever 2 — `additionalStyles` per component (inline styles set inside the IIFE)

What it is: the `defaultTheme.additionalStyles.{Button,Card,...}` object, mutated via wrapper patch on the IIFE's defaultTheme. Lit's `styleMap` on each component reads it and emits `style="..."` on the inner styled element.

What it reaches: the inner element directly (button, section, dialog, etc.), no scope problem.

What it can do: set static styling per component class. Read CSS custom properties via `var()` (resolves at use-time).

What it CAN'T do: respond to pseudo-classes (no `:hover` etc. — `styleMap` is inline only). Differentiate per-instance state without the wrapperPatch + `var()` channel.

Examples in this codebase: every `additionalStyles.{Button,Card,TextField,Slider,Modal,Row}` block in the host file. The custom-property channel (`additionalStyles.Button = { background: 'var(--a2g-btn-bg)' }`) is what makes state-driven theming possible *when combined with* Lever 5.

### Lever 3 — Document-level `@font-face` and global resources

What it is: `@font-face { ... }` declarations on body's `<style>`. Document-level font registration is global and pierces shadow DOM via the document's font set.

What it reaches: any element using the registered family, regardless of shadow boundary depth.

What it can do: bundle and use custom fonts (Material Symbols, JetBrains Mono, etc.). Same pattern works for any "global resource" registered at document level (CSS counter-style, etc.).

What it CAN'T do: scope rules to specific elements (those rules need Lever 5).

Examples in this codebase: `@font-face` for Material Symbols Outlined.

### Lever 4 — `wrapperPatch` (mutate Lit class behavior post-define)

See "The wrapperPatch lever" section above. This is the trust-boundary-clean lever for "I need to change how the renderer behaves" or "I need to inject something into the renderer's shadow."

### Lever 5 — Shadow-scoped `<style>` injection

What it is: a `<style>` element appended to a renderer component's `shadowRoot` via Lever 4 (typically `firstUpdated` hook).

What it reaches: any element inside that specific shadow tree. The injected stylesheet sees the same scope as the IIFE's own static styles.

What it can do: pseudo-classes (`:hover`, `:focus-within`, `:active`), attribute selectors (`[data-primary]`), descendant selectors, anything CSS supports — within scope.

What it CAN'T do: cross to a different shadow tree without a separate injection. The surface tree lives in `a2ui-surface.shadowRoot`. Markdown content lives in each `a2ui-text.shadowRoot`. They're independent scopes.

Examples in this codebase: `A2G_APP_SHADOW_CSS` injected into `a2ui-surface.shadowRoot`, `A2G_TEXT_SHADOW_CSS` injected into each `a2ui-text.shadowRoot`. Both ride the wrapperPatch pattern (Lever 4) to land at the right time.

### Lever decision flow

When you have a styling or behavior question, run through this:

1. Is it a token (color, spacing, font)? → **Lever 1** (custom property on body).
2. Is it static per-component styling that doesn't change with state? → **Lever 2** (`additionalStyles`).
3. Is it a global resource (font, counter)? → **Lever 3** (`@font-face` etc.).
4. Is it state-driven (`:hover`, `[data-*]`, `:focus`) OR a descendant selector? → **Lever 5** (shadow `<style>` injection), reached via **Lever 4** (`wrapperPatch`).
5. Is it a behavior change, missing attribute reflection, or post-render side effect? → **Lever 4** alone.

If none fit, you're proposing to break the trust boundary. Stop and reconsider.

## Diagnostic kit — copy-paste-ready

Save yourself 30 minutes when you next investigate a CSS / shadow / Lit-component question. Drop this in `src/a2glimpse-host.html` between the IIFE and the wrapperPatch scripts, run, capture stdout. Remove when done (or gate on `?diag=1` URL param if you want it permanent).

```html
<script id="a2g-diag-temporary">
// Diagnostic kit — three-round CSS cause isolation + ancestry walk.
// Routes via window.glimpse.send({error:{message:...}}) — Swift forwards
// `error` payloads to stdout. Capture with: `... 2>&1 | grep DIAG`.
setTimeout(() => {
  const findEl = (root, tag, acc = []) => {
    (root.querySelectorAll ? root.querySelectorAll('*') : []).forEach(el => {
      if (el.tagName === tag) acc.push(el);
      if (el.shadowRoot) findEl(el.shadowRoot, tag, acc);
    });
    return acc;
  };
  const walk = (el) => {
    const chain = []; let n = el;
    while (n && chain.length < 15) {
      chain.push(n.nodeType === 11 ? '#shadow-of-' + (n.host?.tagName || '?')
                : n.nodeType === 9 ? '#document'
                : n.tagName);
      n = n.parentNode || n.host;
    }
    return chain.join(' < ');
  };
  const TAG = 'A2UI-BUTTON';   // change per investigation
  const PROBE = '--a2g-btn-bg'; // change per investigation
  const SELECTOR = 'a2ui-button[data-primary]'; // change per investigation

  const out = findEl(document, TAG).map(el => ({
    id: el.id,
    matches: el.matches(SELECTOR),
    probe: getComputedStyle(el).getPropertyValue(PROBE).trim(),
    ancestry: walk(el)
  }));
  window.glimpse?.send?.({ error: { message: 'DIAG ' + JSON.stringify(out) } });
}, 2000);
</script>
```

Read the output:

- **All `matches: false`** → selector is wrong, attribute isn't set, or element isn't found. Round 1 fail.
- **`matches: true`, `probe: ''`, ancestry contains `#shadow-of-*`** → shadow-scoping issue. Use Lever 5.
- **`matches: true`, `probe: <unexpected value>`** → cascade / specificity issue. Trace declarations.
- **`matches: true`, `probe: <expected value>`, but visual is wrong** → not a CSS-matching issue. Look at inline styles, browser defaults, or the styled element's own shadow.

Run again with different `TAG` / `PROBE` / `SELECTOR` to triangulate.
