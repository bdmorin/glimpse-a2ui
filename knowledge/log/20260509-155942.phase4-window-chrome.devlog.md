# Phase 4 — Window Chrome Polish

## Author
Claude Opus 4.7 (1M context), worktree agent-ae33ddb8eee87dfba.

## Context
Phase 4 row of `knowledge/20260509-140000.polish-and-hardening-plan.plan.md`. Bring the production a2glimpse window in line with Brian's reference image: traffic lights present, no separate app-name title strip, content flowing under the titlebar so the appliance reads as a modern macOS surface rather than upstream Glimpse's stock chrome. Dark mode listed as STRETCH.

## Did
1. **Unified macOS titlebar (Swift, production-mode only).** In `src/a2glimpse.swift::setupWindow()`, added a gated block right after `window.title = config.title`:
   ```swift
   if !config.testMode && !config.frameless {
       window.titleVisibility = .hidden
       window.titlebarAppearsTransparent = true
       window.styleMask.insert(.fullSizeContentView)
   }
   ```
   Gate rationale: `--test-mode` keeps standard chrome (Phase 1 contract — visual goldens depend on deterministic geometry). `--frameless` is borderless already; the unified-titlebar attributes are no-ops on that style mask but keeping the gate explicit avoids future surprises if we add a different chrome mode.

2. **Host-page top-padding compensation (CSS, production-mode only).** In `src/a2glimpse-host.html`, appended:
   ```css
   body:not([data-test-mode]) { padding-top: 48px; }
   ```
   Rationale: body already had `padding: 20px 24px`. Traffic-light area is ~28px tall; 48 = 20 + 28 puts content visually under the lights with the existing 20px breathing room intact. Test-mode keeps the original 20px (selector excludes `[data-test-mode]`) so test fixtures render at the same coordinates as before.

3. **Default surface size 800×600 → 480×320.** Upstream Glimpse defaulted to 800×600 because surfaces were arbitrary HTML. A2UI surfaces are denser and smaller (single Card, modal, compact form). 480×320 matches the Phase 1 test-mode default — same dimensions used in every visual fixture. Callers can still override with `--width`/`--height`.

4. **Content padding split decision: kept in CSS.** Phase 3 added `padding: 20px 24px` to the body. Default of the orchestrator brief was "leave it in CSS unless you find a concrete reason to move it." No reason found. Single source of truth (host CSS) for visual chrome makes sense — Swift's job is window geometry, not interior layout.

## Considered & rejected
- **`env(titlebar-area-height)` instead of fixed 28px.** Looked into it; WKWebView support is unclear and the `titlebar-area-*` env vars are PWA-titlebar-specific (not the Cocoa fullSizeContentView path). Fixed 48px is deterministic and matches the macOS HIG traffic-light region. Revisit if WebKit ever standardizes a titlebar inset for embedded WebViews.
- **Move padding to Swift (WKWebView contentInsets).** Considered; rejected. Two sources of truth (Swift inset + CSS padding) would race and is harder to reason about in the test-mode/production split. CSS owns interior layout cleanly.
- **Dark mode (stretch).** DEFERRED — see "Dark mode deferral" below.

## Dark mode deferral
After scoping the work, dark mode is not a clear ~30-minute path:

1. **Token surface is large.** The host CSS defines ~12 `--md-sys-color-*` and 2 `--md-sys-elevation-*` custom properties tuned for a light Material 3 palette (primary `#2563eb`, surface `#ffffff`, neutrals `#d0d7de`/`#e6eaee`/etc). A credible dark scheme requires designing matching tokens — not just inverting — to keep contrast ratios and Material 3 elevation conventions intact. That's design work, not a tweak.
2. **Live-toggle wiring is real plumbing.** Need: appearance-change observer in Swift (`NSApp.observe(\.effectiveAppearance, ...)` or `viewDidChangeEffectiveAppearance`), a JS bridge call to update `body.dataset.colorScheme`, and CSS rules scoped on that attribute. Plus a test-mode pin (light only) to keep goldens deterministic. Each piece is small but the integration test surface is broader than the unified-titlebar slice.
3. **Risk to visual goldens.** Even with a test-mode pin, any dark-mode CSS that mis-scopes will quietly shift goldens at next bless cycle. Adding it carefully is a slice on its own.
4. **MUST is the headline.** Brian's brief explicitly said "if dark-mode work isn't on a clear path to completion, DEFER." The unified-titlebar MUST is shipped, tested, and verified. Shipping dark mode half-done would be worse than shipping it later cleanly.

Recommendation: dedicate a separate Phase 4b slice to dark mode with a designed token pair, an explicit listener path, and a fixture pair (light/dark goldens). Not in scope for this PR.
