## Architectural decisions

- **Stack over tabs over multi-window.** Tabs reuse `<a2ui-tabs>` but defeat the load-bearing use case ("status ticks while diff awaits approval"). Multi-window means owning N OS windows in Swift — out of scope for v1. Stack is the simplest model that supports the use case.
- **Auto-grow is automatic, not agent-driven.** The agent never decides "how tall should this be"; the page measures content and tells Swift. Mid-implementation correction from Brian made this explicit. The `resize` tool stays as an optional override but the skill description repositions it as "you only need this to widen the window or pin a fixed size."
- **`__a2glimpse_debug` is a convention, not enforcement.** The bridge does not gate the surfaceId; the host wrapper just gives it distinct chrome. The agent renders normal A2UI components into it. This avoids carving a privileged channel.
- **`tick` counter pattern.** Lit reactivity normally watches property changes. Surfaces inside the IIFE's processor are signal-driven and self-update; what we needed was a way to force *our* `A2GlimpseApp.render()` to re-evaluate when the visible-stack changes. A monotonic `tick` bumped on every `dispatchA2ui` is the lowest-effort lever — it's a property, Lit re-renders on any property change, and we don't have to track per-surface dirtiness.
- **Test-mode `min-height: 100vh` gate.** Removing the `min-height: 100vh` from `:host` was necessary for production auto-shrink-to-content. But test-mode goldens depend on the host filling the viewport. Solution: set `style.minHeight = '100vh'` in `connectedCallback` when `body[data-test-mode]` is present. This keeps the visual baseline byte-stable for design-track while production gets the new behavior.

## Considered / rejected

- **Per-tab presentation (`<a2ui-tabs>` reuse).** Rejected — see above. Defeats the parallel-status use case.
- **Multi-window (one OS window per surface).** Rejected for v1. Possible v3 work.
- **Required `surfaceId` on `await_action`.** Rejected — too strict; agents that don't care about ordering would have to invent disambiguation. Optional filter with unfiltered fallback covers both modes.
- **A bridge-side debug rendering tool that bypasses the validator.** Rejected — would punch a hole in the trust boundary just for ergonomics. The convention-based reserved surfaceId achieves the same UX without breaching.
- **Auto-grow width as well as height.** Rejected — vertical stack means height is the natural growth axis. Width changes would feel jittery as text reflows.
- **Re-blessing all visual goldens during the slice.** Rejected once Brian flagged the design-track parallelism. The host hash will rotate again when design-track lands; one re-bless at merge is cheaper than two side-by-side blesses now.

## Failures and recoveries

- **TypeScript parameter properties.** Initial `RingBuffer` had `constructor(private capacity: number)`. Node 22's strip-only TS mode rejects this (`ERR_UNSUPPORTED_TYPESCRIPT_SYNTAX`). Refactored to explicit field + assignment.
- **Smoke regression on `userAction.surfaceId`.** Removing `this.surfaceId` from `A2GlimpseApp` broke the `handleAction` fallback chain — surface IDs started arriving as `"@default"` instead of `"smoke"`. Root cause: `detail.sourceComponent.surfaceId` propagation isn't bullet-proof, and the prior code papered over that with the App's tracked single id. Fix: when exactly one surface is visible, use it as the fallback; otherwise `"@default"`.
- **Visual goldens mismatch at the new host hash.** Loss of `:host { min-height: 100vh }` made the surface card collapse to content-height in test mode, diverging from the prior baseline. Eyeballed via `--update --only button-only` (gated to a single fixture) → confirmed visible drift → gated `min-height` on `body[data-test-mode]` → did NOT bless the rest. Design-track's `0ff4d09524c9` baseline left untouched.
