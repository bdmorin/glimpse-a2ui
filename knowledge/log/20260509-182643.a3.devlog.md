## Architectural decision

**Skill-only, no custom tool wrapper.** Brian flagged the question explicitly. Three options on the table:

1. Skill teaches `mcporter call a2glimpse.<tool>` directly via Bash. Simplest. Zero new surface area.
2. Custom Claude Code tool that wraps mcporter calls. Adds a layer; doubles the surface area to teach (now you must teach the wrapper AND the underlying tools). No win.
3. New native MCP server bundling skill + tool. Brian explicitly rejected this pattern (snap-happy retrospective): native MCP wiring when an mcporter shim is already on the table is ceremony theater.

Picked option 1. The MCP bridge already exposes the right surface (`surface_update`, `data_model_update`, `begin_rendering`, `await_action`, `close`, etc.) — the skill teaches the agent to compose those, not invent a new abstraction. If empirical use shows agents fumbling the JSONL composition repeatedly, revisit by adding inline templates as scripts in `skills/a2glimpse/scripts/` — but don't pre-emptively wrap.

## Considered / rejected

- **Listing every A2UI component in the skill.** Rejected — there are ~30 and agents face seven interaction shapes. The catalog doc is for completeness; the skill is for triage.
- **Inline-prosing each MCP tool.** Rejected — single compact reference table once. Repeating "this tool forwards X" per pattern would 3x the length.
- **Splitting patterns into per-file references** (`skills/a2glimpse/references/confirm.md`, etc.). Rejected — the seven patterns total ~200 lines. Splitting forces the agent to navigate a directory for content that fits inline. Anthropic's progressive-disclosure guidance kicks in around 500 lines; we're well under.
- **A separate Codex `skills.md`.** Rejected — Codex reads `SKILL.md` with the same two frontmatter fields. Single file is the cross-platform path.
- **Separate "for humans" README at `skills/a2glimpse/README.md`.** Rejected. The SKILL.md IS the doc; humans (including Brian) can read it. Doubling the artifact invites drift.

## Reference synthesis (what the four URLs taught me)

- **Anthropic** — `description` field is the trigger; ≤1024 chars; third-person; under-500 line body; one-level-deep references; concrete examples beat prose; gotchas earn their tokens. Eval-first iteration loop with Claude A (designer) / Claude B (consumer).
- **Codex** — Same SKILL.md format; constrained metadata budget (~8k chars across the skill list); explicit + implicit invocation modes; optional `agents/openai.yaml` for UI; doesn't address Claude compat directly but the format is shared.
- **agentskills.io** — "Add what the agent lacks, omit what it knows." Gotchas sections are the highest-value content. Provide defaults not menus. Match specificity to fragility per-section. Procedures > declarations.
- **Vercel react-best-practices** — Structural exemplar: impact labels (CRITICAL/HIGH/etc.) for scanning, "When to use / When NOT to use" sub-sections, "Incorrect:" / "Correct:" code-pair pattern. Heavier than Anthropic's 500-line guideline, but it's a reference doc not a workflow skill.

All four agreed on the load-bearing principles. Codex constrains harder on metadata budget; Vercel ignores the line guideline (different use case). Picking the strictest interpretation everywhere gives cross-platform compatibility.
