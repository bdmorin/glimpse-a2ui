## Goal

Author a runbook for re-vendoring the inlined `@a2ui/lit` IIFE in
`src/a2glimpse-host.html` when google/A2UI ships v0.8.x bugfixes, plus a
pinning ledger to track which bundles have shipped.

## What I did

- Verified the upstream layout in `~/src/github.com/google/A2UI`. v0.8 source
  lives at `renderers/lit/src/0.8/`; current `package.json` is on 0.9.3 with
  a `wireit`-driven `npm run build`. Tag `v0.8` resolves to commit
  `19919ef4c8ad3185867f70386fa4669284d7714c`. The exact bundle output path
  inside `dist/` was NOT verified — the runbook flags it as a first-operator
  capture point.
- Inspected `src/a2glimpse-host.html` and identified every region (host
  styles, markdown-engine importmap shim, vendored IIFE 208–11077, host-ready
  shim, Tabs prototype patch, Modal auto-open patch). Documented which are
  ours vs upstream, with the rule "only the IIFE is replaced wholesale".
- Read `test/visual.mjs` and confirmed snapshot dir is keyed on the SHA-256
  of the **whole** `src/a2glimpse-host.html` truncated to 12 hex chars
  (current: `0ff4d09524c9`). Wrapper edits invalidate goldens just as much
  as IIFE swaps. Captured this in the re-bless protocol section.
- Computed two hashes for the bundle currently in tree:
  - File hash: `0ff4d09524c9c73b743a0ed4911b89854f672853a0dbe6484e2add2e10c864d7`
    (truncated `0ff4d09524c9`, matches the existing `test/__snapshots__/` dir).
  - IIFE-only (lines 209–11077): `e02f68273f5b0cc27099b7434d6dd4f0951e75815700e7619d42ad6acb97a9ad`
    (truncated `e02f68273f5b`).
  - Recorded both as the initial bundle-pins ledger row, with upstream sha
    marked `<unknown — capture on first re-vendor>` because the original
    paste predates this fork's hygiene. The IIFE-only hash gives us a stable
    identity for "the bytes we currently ship" independent of wrapper edits.

## Decisions and rationale

- **Pinning policy = v0.8.x only.** Per HANDOFF, v0.9 is a spec major pivot
  and explicitly out of scope. The runbook says so loudly.
- **Trust-boundary diff is mandatory before paste, not after.** Catching a
  new `eval`/dynamic-import/network primitive after blessing goldens is too
  late. Putting the diff step at 4.3, before swap-in (4.4), makes "stop and
  escalate" the actual blocking move.
- **Bless on a re-vendor branch, never on main.** Ensures main's
  previous-hash goldens stay valid until merge, which gives reviewers a
  before/after pair.
- **Three-commit shape on the re-vendor branch.** Splitting "swap IIFE",
  "re-bless goldens", and "drop workarounds" into separate commits makes
  the bless reviewable independent of the bundle bytes (which a human
  cannot meaningfully review at 11k lines anyway). The trust review is the
  diff check at 4.3; the goldens are the visual review.
- **Workaround re-verification is mandatory.** Code we don't carry is code
  we don't maintain. Tabs prototype patch and Modal auto-open patch are
  both candidates for removal on every re-vendor; the runbook walks through
  each one with how-to-test-removal.
- **Confidence: LIKELY, not VERIFIED.** I have not actually re-vendored
  anything. The build path is unknown. The runbook's "First-time gotchas
  to capture" section explicitly says: edit in place on first run, then
  bump to VERIFIED.

## What I did NOT do

- Did not run `npm run build` upstream to verify the bundle output path.
  Out of scope for an unproven runbook — first operator captures it.
- Did not attempt to identify the upstream sha the current bundle was
  built from. Could in principle bisect by content-hashing historical
  builds, but the value isn't worth the cost; first re-vendor establishes
  a known-good baseline anyway.
- Did not change anything in `src/a2glimpse-host.html`. This slice is
  documentation only.

## Competing hypothesis

The runbook assumes upstream's `npm run build` emits something IIFE-shaped
or that we can wrap their ESM into an IIFE on swap-in. If upstream switches
to a build artifact that requires a real bundler step on our side, the
"hand-paste" workflow stops working and this runbook needs a build-step
section. The first-time gotchas section flags this explicitly.
