# Slice B1 devlog

## What

Drafted three filing-ready GitHub issue bodies under `knowledge/filings/`, plus an index README and an INDEX.md update. Each filing follows the recon-derived house style (Symptom → Reproduction → Root cause → Spec reference → Related issues → Suggested fix), uses backticked code identifiers in titles ≤90 chars, and includes paste-ready JSONL repros.

## Source provenance

Local A2UI clone at `~/src/github.com/google/A2UI` provided the upstream source for grounding:

- `renderers/lit/src/0.8/ui/tabs.ts` line 67 — confirms `classMap(this.theme.components.Tabs.element)` is the crash site.
- `renderers/web_core/src/v0_8/types/types.ts` lines 127–134 — confirms `Tabs.element` is declared required at the type level.
- `renderers/lit/a2ui_explorer/src/theme.ts` lines 339–343 — confirms the in-tree explorer theme DOES include `element: {}`, which is why the gap is silent during in-tree development.
- `renderers/lit/src/0.8/ui/multiple-choice.ts` line 39 — confirms `accessor variant` (not `type`) on the element.
- `renderers/lit/src/0.8/ui/multiple-choice.ts` lines 288–304 — confirms the `getCurrentSelections` fall-through that crashes on empty literalArray.
- `renderers/web_core/src/v0_8/schema/common-types.ts` lines 283–316 — confirms the schema accepts `literalArray` and the field name is `type` (line 314).

Vendored bundle in `src/a2glimpse-host.html` provided the consumer-side evidence:

- Line 8265: `.variant=${node.properties.variant}` — confirms catalog binding reads `variant` not `type`.
- Line 10314: `${e10(this.theme.components.Tabs.element)}` — confirms the published bundle still ships the dereference.
- Lines 11119–11144: existing prototype patch on `<a2ui-tabs>.update`, copied verbatim into the Tabs filing's "Suggested fix" as the consumer-side workaround anchor.

## Decisions

- **Used `type: filing`** as a new common-core type (per brief instruction). Frontmatter includes `issue_title` and `issue_url_target` so Brian can paste with one copy.
- **Did not include `[@a2ui/lit@0.8.x]` SHA in titles.** No definitive consumer-bundle SHA available in `src/a2glimpse-host.html` header; package.json reports `0.8.0` but that's the a2glimpse fork version, not the renderer bundle hash. Left titles as `[@a2ui/lit] ...` and let the body name the version family ("Affects @a2ui/lit for A2UI v0.8."). House style favors precision when known and absence when not.
- **Cross-referenced the three filings to each other** in each "Related issues" section. After filing, real issue numbers can replace the prose link.
- **Cross-referenced closed #574** in filings 2 and 3 as historical context only — no claim of equivalence.
- **Left a single `<verify exact source path>` placeholder** in filing 2 for the catalog binding location in upstream Lit source. Found the binding cleanly in the vendored bundle (line 8265); spent reasonable time looking for the equivalent in `renderers/lit/src/0.8/...` but it's likely generated/in a catalog file we didn't pin down. Surfacing it as `<verify ...>` is the honest move per the brief.
