# `[@a2ui/lit] MultipleChoice.getCurrentSelections throws when selections.literalArray is []`

## Symptom

A spec-valid `MultipleChoice` whose `selections` is `{ "literalArray": [] }` (no preselected values, expressed as a literal rather than a path binding) crashes the renderer at first paint. The exception originates inside `getCurrentSelections()`, which calls `processor.getData(component, undefined, surfaceId)` because it never branches on the `literalArray` shape — `selections.path` is undefined and `Array.isArray(this.selections)` is false (it's an object), so it walks the path branch with no path. Affects `@a2ui/lit` for A2UI v0.8.

## Reproduction

Spec-valid v0.8 message:

```json
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"mc","componentProperties":{"MultipleChoice":{"options":[{"label":{"literalString":"A"},"value":"a"}],"selections":{"literalArray":[]}}}}]}}
{"beginRendering":{"surfaceId":"s1","root":"mc","catalogId":"a2ui-v0.8"}}
```

Throws at first render. Switching `selections` to `{ "path": "/picked" }` and providing a `dataModelUpdate` for `/picked` avoids the crash; switching to a non-empty `literalArray` (e.g. `["a"]`) also avoids the crash because… (see root cause).

## Root cause

`renderers/lit/src/0.8/ui/multiple-choice.ts` lines 288–304:

```ts
getCurrentSelections(): string[] {
  if (Array.isArray(this.selections)) {
    return this.selections;
  }

  if (!this.processor || !this.component) {
    return [];
  }

  const selectionValue = this.processor.getData(
    this.component,
    this.selections.path!,
    this.surfaceId ?? A2uiMessageProcessor.DEFAULT_SURFACE_ID,
  );

  return Array.isArray(selectionValue) ? (selectionValue as string[]) : [];
}
```

`this.selections` is the raw schema object `{ literalArray?: string[]; path?: string }`. `Array.isArray(this.selections)` is always false (it's an object, not an array). The function then walks the path branch unconditionally and passes `this.selections.path!` — undefined — into `processor.getData`. The non-null assertion (`!`) silences the TypeScript guard; at runtime, `getData` is called with an undefined path and throws.

The empty-array case is the only shape that reliably hits the bug because nothing else upstream (e.g. the element's own initialization) accidentally normalizes `[]` away. A non-empty `literalArray` may sometimes appear to "work" only because consumers tend to pair it with a path binding too — the bug is that `literalArray` is never honored as a source of truth here regardless of length.

The schema accepts `literalArray` (`renderers/web_core/src/v0_8/schema/common-types.ts` lines 283–293):

```ts
export const MultipleChoiceSchema = z.object({
  selections: z
    .object({
      path: z.string().describe("...").optional(),
      literalArray: z.array(z.string()).optional(),
    })
    .strict()
    .superRefine(exactlyOneKey),
  // ...
});
```

So `selections.literalArray:[]` is a fully spec-valid input that the renderer cannot consume.

## Spec reference

`MultipleChoiceSchema` in `renderers/web_core/src/v0_8/schema/common-types.ts` allows `selections` to be either `{ path }` or `{ literalArray }`, enforced by `superRefine(exactlyOneKey)`. The renderer must handle both shapes for valid surfaceUpdates not to crash.

## Suggested fix

Branch on the `literalArray` shape before falling through to `processor.getData`. Roughly:

```ts
getCurrentSelections(): string[] {
  if (Array.isArray(this.selections)) {
    return this.selections;
  }
  if (this.selections && Array.isArray(this.selections.literalArray)) {
    return this.selections.literalArray;
  }
  if (!this.processor || !this.component || !this.selections?.path) {
    return [];
  }
  const selectionValue = this.processor.getData(
    this.component,
    this.selections.path,
    this.surfaceId ?? A2uiMessageProcessor.DEFAULT_SURFACE_ID,
  );
  return Array.isArray(selectionValue) ? (selectionValue as string[]) : [];
}
```

`toggleSelection` / `#setBoundValue` will also need to no-op (or rewrite) the literal-array case — selecting into a literal array has no obvious "where do I write back" answer; either keep the literal as initial-state-only (read-only after first paint) or reject the shape at schema time. Either decision is fine for downstream consumers; the load-bearing requirement is that the empty-array case stops throwing at first paint.

A regression test asserting `MultipleChoice` with `selections.literalArray:[]` renders without throwing would lock this in.
