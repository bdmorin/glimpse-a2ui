# `[@a2ui/lit] Tabs.render: blank component when defaultTheme.components.Tabs.element is missing`

## Symptom

Rendering an A2UI v0.8 surface that contains a `Tabs` component produces an empty shadow root. The host element mounts, but the inner `<div id="buttons">` and its tab buttons never appear; the tab content slot also stays empty. Sibling components (Card, Text, Button, MultipleChoice) on the same surface render normally, so the failure is local to `<a2ui-tabs>`. Reproduces against `@a2ui/lit` v0.8 builds whose `defaultTheme.components.Tabs` omits the `element` key.

## Reproduction

A2UI v0.8 message that triggers the failure:

```json
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"root","componentProperties":{"Tabs":{"titles":[{"literalString":"One"},{"literalString":"Two"}],"selected":0}}}]}}
{"beginRendering":{"surfaceId":"s1","root":"root","catalogId":"a2ui-v0.8"}}
```

With a `defaultTheme.components.Tabs` that defines `container`, `controls.all`, `controls.selected` but not `element` (the shape that ships in the consumer-facing bundle we pulled), the call into `Tabs.render` throws `TypeError: cannot convert undefined or null to object` from `Object.keys` inside lit-html's `classMap` directive. The unhandled rejection mid-update causes Lit to abort the shadow-DOM commit, so the component renders blank.

Grep against the published renderer bundle confirms the offending dereference:

```bash
grep -n "this.theme.components.Tabs.element" a2ui-host.html
# 10314:    return b2`<div id="buttons" class=${e10(this.theme.components.Tabs.element)}>
```

## Root cause

`renderers/lit/src/0.8/ui/tabs.ts` (line 67) renders the buttons row with:

```ts
return html`<div id="buttons" class=${classMap(this.theme.components.Tabs.element)}>
```

`classMap` is `lit-html`'s class directive, which iterates `Object.keys(...)` over its argument. When `theme.components.Tabs.element` is `undefined`, `Object.keys(undefined)` throws — the exception propagates out of `render()`, Lit's update loop aborts, and the component's shadow tree is left empty. There is no defensive default in `tabs.ts` and no schema-time check that ensures `element` exists on the `Tabs` theme entry.

The TypeScript surface in `renderers/web_core/src/v0_8/types/types.ts` (lines 127–134) declares `Tabs.element` as required:

```ts
Tabs: {
  container: Record<string, boolean>;
  element: Record<string, boolean>;
  controls: {
    all: Record<string, boolean>;
    selected: Record<string, boolean>;
  };
};
```

…but no runtime guard enforces it, and at least one shipped/vendored `defaultTheme` instance lacks the `element` key. (For comparison, the in-repo explorer theme at `renderers/lit/a2ui_explorer/src/theme.ts` lines 339–343 includes `element: {}` and renders correctly — which is exactly why the gap is silent during in-tree development.)

## Spec reference

The A2UI v0.8 component spec for `Tabs` (`specification/...`) defines a renderable `Tabs` element; it does not require consumers to construct a theme to use the component. A consumer providing a valid surfaceUpdate with valid `Tabs` properties and the renderer's own `defaultTheme` should not produce a blank component.

## Suggested fix

Two minimal options (either is acceptable, the first is preferred because it preserves the typed contract):

1. Ensure every shipped `defaultTheme.components.Tabs` includes an `element: {}` key. This matches the existing TS interface in `web_core/src/v0_8/types/types.ts` and the explorer theme.
2. Make `Tabs.render` defensive: substitute `this.theme.components.Tabs.element ?? {}` at the `classMap` call site (and similarly for `controls.all` / `controls.selected`).

For downstream consumers stuck on a vendored bundle with the gap, the workaround we ship is a wrapper-side prototype patch on `<a2ui-tabs>.update`:

```js
customElements.whenDefined('a2ui-tabs').then((TabsCtor) => {
  const orig = TabsCtor.prototype.update;
  TabsCtor.prototype.update = function (changedProps) {
    const t = this.theme;
    if (t && t.components && t.components.Tabs) {
      const tabs = t.components.Tabs;
      if (!tabs.element) tabs.element = {};
      if (!tabs.controls) tabs.controls = { all: {}, selected: {} };
      else {
        if (!tabs.controls.all) tabs.controls.all = {};
        if (!tabs.controls.selected) tabs.controls.selected = {};
      }
    }
    return orig.call(this, changedProps);
  };
});
```

This unblocks rendering without modifying the IIFE; an in-tree fix in `tabs.ts` (or in the shipped defaultTheme) would let consumers drop the patch.
