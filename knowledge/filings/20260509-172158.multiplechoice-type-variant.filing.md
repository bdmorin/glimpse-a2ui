# `[@a2ui/lit] MultipleChoice silently ignores spec field type (element reads .variant)`

## Symptom

A surfaceUpdate that uses the spec-valid `MultipleChoice.type` field has no visual effect — the component always renders in `checkbox` mode (the element's hard-coded default), regardless of whether the agent sets `type:"checkbox"` or `type:"chips"`. No validation error is raised; the field is silently dropped during property binding. Affects `@a2ui/lit` for A2UI v0.8.

## Reproduction

Spec-valid v0.8 message asking for chips mode:

```json
{"surfaceUpdate":{"surfaceId":"s1","components":[{"id":"mc","componentProperties":{"MultipleChoice":{"type":"chips","options":[{"label":{"literalString":"A"},"value":"a"},{"label":{"literalString":"B"},"value":"b"}],"selections":{"path":"/picked"}}}}]}}
{"dataModelUpdate":{"surfaceId":"s1","contents":[{"path":"/picked","contents":["a"]}]}}
{"beginRendering":{"surfaceId":"s1","root":"mc","catalogId":"a2ui-v0.8"}}
```

The component renders as a checkbox list, not chips. Replacing `"type":"chips"` with `"variant":"chips"` (a non-spec field) makes chips render — confirming the field-name mismatch is the cause.

Grep against the published renderer bundle:

```bash
grep -n "MultipleChoice" a2ui-host.html | head
# 8252:          case "MultipleChoice": {
# 8265:            .variant=${node.properties.variant}
```

## Root cause

The A2UI v0.8 schema defines the field as `type` — `renderers/web_core/src/v0_8/schema/common-types.ts` line 314:

```ts
export const MultipleChoiceSchema = z.object({
  // ...
  maxAllowedSelections: z.number().optional(),
  type: z.enum(['checkbox', 'chips']).optional(),
  filterable: z.boolean().optional(),
});
```

The Lit element exposes the same concept under a different name — `renderers/lit/src/0.8/ui/multiple-choice.ts` line 39:

```ts
@property()
accessor variant: 'checkbox' | 'chips' = 'checkbox';
```

The catalog binding in the Lit renderer reads from `node.properties.variant` rather than `node.properties.type` (vendored bundle line 8265, mirroring the in-tree binding in `renderers/lit/src/0.8/...` — `<verify exact source path>`):

```ts
case "MultipleChoice": {
  // ...
  .variant=${node.properties.variant}
  // ...
}
```

So a spec-valid `MultipleChoice.type` parses cleanly through the schema but never reaches the element. The runtime resolver does not remap `type` → `variant`. The result is silent property loss — the worst kind, because nothing surfaces an error and the agent has no signal that the field was ignored.

## Spec reference

`MultipleChoiceSchema` in `renderers/web_core/src/v0_8/schema/common-types.ts` is the authoritative v0.8 schema and defines the field as `type`. The mirroring component documentation in `specification/...` likewise names the field `type`. The Lit element's `variant` accessor is an internal name choice; either the element should accept the spec name or the catalog binding should remap it.

## Suggested fix

Two paths, in order of preference:

1. Rename the element's `variant` accessor to `type` in `renderers/lit/src/0.8/ui/multiple-choice.ts` and update its catalog binding to `.type=${node.properties.type}`. This collapses spec and element to one name and removes the entire class of "silent drop" failures.
2. If the element name must remain `variant` for compatibility, remap inside the catalog binding: `.variant=${node.properties.type}`. This keeps the spec field as the source of truth and isolates the rename to one site.

In either case, please add a unit test in `renderers/lit/tests/...` that constructs a `MultipleChoice` surface with `type:"chips"` and asserts the rendered element actually presents chips — that's the regression that would have caught this.
