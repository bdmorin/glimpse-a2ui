# Desktop-Pet Sprite Ecosystem

## TL;DR

- **OpenAI Codex defines the format** (de facto, via its desktop app's "Hatch Pet" skill / `/pet` slash command) and emits pets into `~/.codex/pets/<slug>/`.
- **Petdex** (`crafter-station/petdex`, https://petdex.crafter.run/) is a one-week-old community registry + CLI + R2 CDN distributing 1618+ Codex-format pets. MIT-licensed code; fan-art sprites with takedown policy.
- **Openpets** (`alvinunreal/openpets`) is a downstream Electron consumer that re-uses the *exact same* 1536×1872 / 8×9 / 9-row layout for non-Codex environments.
- **All three converge on a shared sprite spec** with byte-for-byte identical animation row names (`idle, running-right, running-left, waving, jumping, failed, waiting, running, review`).
- **a2glimpse will adopt the format and "bring your own pet" approach** — read pets from `~/.codex/pets/<slug>/`, do not redistribute, sidestep IP/takedown surface entirely. `dr-glitch` (already on Brian's host, verified plug-compatible) is the default.

## The Format

Sprite spec (verified across all three projects):

| Field | Value |
|---|---|
| Spritesheet dimensions | **1536 × 1872 px** |
| Grid | **8 columns × 9 rows** |
| Frame size | **192 × 208 px** |
| Format | PNG or WebP (WebP preferred — `dr-glitch` is 2.0 MB WebP) |
| Manifest | `pet.json` alongside spritesheet |

**Animation rows (fixed order, row 0–8):**

| Row | Name | Typical use |
|---|---|---|
| 0 | `idle` | default loop, agent quiescent |
| 1 | `running-right` | moving / active state |
| 2 | `running-left` | moving / active state, reversed |
| 3 | `waving` | greeting / startup |
| 4 | `jumping` | celebration / completion |
| 5 | `failed` | error state |
| 6 | `waiting` | blocked / awaiting input |
| 7 | `running` | working / processing |
| 8 | `review` | code-review / consideration mode |

Frame counts per row vary (4–8); animation durations 700–1220 ms per loop in Petdex's reference implementation. Petdex's server-side validator only enforces ≥256×256 and "ideal 1536×1872" — strict grid is **convention**, not enforced. `pet.json` shape (verified from `~/.codex/pets/dr-glitch/pet.json`):

```json
{
  "id": "dr-glitch",
  "displayName": "Dr. Glitch",
  "description": "A tiny alien mad scientist digital pet with manic lab energy and compact chibi proportions.",
  "spritesheetPath": "spritesheet.webp"
}
```

**Codex is the spec owner.** Petdex is distribution. Openpets is a renderer-only consumer. The whole ecosystem is **less than two weeks old** as of 2026-05-09 — Codex's pet skill, Petdex registry, and Openpets all surfaced within days of each other.

## License / IP Posture

- **Petdex code:** MIT.
- **Openpets code:** MIT.
- **Codex itself:** proprietary OpenAI product.
- **Pet sprites:** **NOT covered by Petdex's MIT.** Petdex explicitly disclaims IP rights — sprites are user-submitted fan art, AI-generated by Codex's Hatch Pet skill from user prompts. Many will resemble trademarked characters (Pikachu, Mario, etc.). Per-pet attribution comes from Clerk-verified submitter identity, surfaced as `/u/<handle>` on Petdex profiles.
- **Takedown:** Petdex has a takedown channel. Anyone redistributing pets inherits that surface.

## Risk Flags

1. **"Codex" is OpenAI's brand.** Tying our app to "Codex format" reads as OpenAI-adjacent. Risk of brand confusion / takedown if framed as official. Mitigation: refer to it as "Codex/Petdex sprite format" or just "1536×1872 desktop-pet format" in user-facing copy.
2. **Sprite IP exposure.** Don't redistribute Petdex sprites in our binary or `.app` bundle. Bring-your-own-pet sidesteps this.
3. **R2-only asset hosting.** Petdex pets live on Cloudflare R2; if Petdex disappears, pets 404. The `pets/` directory in their git repo is just `ideas.json` — sprites are NOT in the git tree. Pets installed locally (via `npx petdex install` or Codex generation) survive Petdex outages.
4. **Spec churn.** Project is one week old and pushing daily. The 9-row layout is hard-coded in `src/lib/pet-states.ts` but row 10 could appear. Build for "row N is named X," not a fixed count.
5. **Attribution.** If we ever build a UI listing pets, surface the submitter handle (Petdex does this — `/u/<handle>` is first-class).
6. **No spec doc.** There's no formal RFC. The spec is the union of `crafter-station/petdex/packages/petdex-cli/README.md` + `src/lib/pet-states.ts` + the AI-generated outputs of Codex's Hatch Pet skill. If you need a stable spec, vendor a snapshot.

## a2glimpse Integration Plan

### Approach: bring your own pet

When `a2glimpse` is launched without a stdin pipe (Finder-launch — currently shows the C1 NSAlert), enter **menubar mode**:

1. Read pets from `~/.codex/pets/` (the same directory Codex Hatch Pet emits to and `npx petdex install` writes into). Iterate available `<slug>/pet.json` entries.
2. Default pet: `dr-glitch` if present; otherwise the first pet found; otherwise show a "no pets installed — try `npx petdex install <slug>`" alert and quit.
3. Render via `NSStatusItem` (menubar tray icon) + an optional frameless transparent always-on-top `NSWindow` for the pet body. Animation loop: `Timer` at ~8 fps cycling row-0 (idle) frames cropped from the sheet via `NSImage(cgImage: cgImage.cropping(to:))`.
4. Tray menu: pet picker (lists `~/.codex/pets/` entries), "Open Petdex…" (`open https://petdex.crafter.run`), "Quit."
5. Default to row 0 (idle) for v1. Future: map a2glimpse runtime states (pipe disconnected, MCP active, error) onto rows 5/7/8.

### What we explicitly do NOT do

- **Do not redistribute pet sprites.** No pets in the git repo, no pets in the `.app` bundle, no fallback bundled pet.
- **Do not vendor Petdex CLI.** Users get pets via `npx petdex install` or Codex itself; we read whatever's there.
- **Do not embed Electron / openpets code.** Electron at 150 MB defeats our zero-dep Swift binary aesthetic. We borrow the format spec only.
- **Do not invent our own format.** Adopting Codex/Petdex format means a2glimpse pets and Codex pets are interchangeable — drop a new pet into `~/.codex/pets/`, both apps see it. That's the whole point of converging.

### Effort estimate

1–2 sessions:

- **Session 1:** ~50 lines of Swift in `src/a2glimpse.swift` adding a `MenubarPetMode` activated on `isatty(0) == 1`. Replaces the C1 NSAlert. Manifest discovery, sprite loading, idle-loop renderer, basic tray menu.
- **Session 2 (optional polish):** runtime-state-to-row mapping, "Open Petdex" menu, About dialog, settings persistence (last-selected pet).

### Trust boundary

Pet loading is read-only file I/O on a user-owned directory. No code execution from sprite metadata. `pet.json` is a small, validated JSON shape (id, displayName, description, spritesheetPath). The image loader is `NSImage(contentsOfFile:)` against PNG/WebP — same surface as any image-displaying Mac app. No new trust-boundary surface.

## Open Questions

- **Codex itself — is the Hatch Pet skill open-source?** Recon didn't establish whether the generation pipeline is public or proprietary inside the Codex desktop app. If proprietary, we depend on OpenAI keeping pets emitted in this format.
- **Spec stabilization.** When (if) the 9-row layout gets versioned, who owns the version number? Today there is none. We should pin against `crafter-station/petdex@<sha>` for our reference and revisit on first compatibility break.
- **Multi-pet rendering.** v1 is single-pet. Multi-pet (a Tamagotchi-style aquarium) is a future-future thing — not in scope until A1-A3 ship.
