# glimpse-a2ui Fork Architecture

> **This is an assessment, not a built thing.** No code in the repo reflects this yet. Read the two `*.knowledge.md` siblings first for grounded baseline facts.

## Thesis

Glimpse is **90% of an A2UI client** already. The JSONL stdio framing maps 1:1 onto A2UI's streaming JSON wire format. The native WebView shell is the right rendering surface. The fork's job is to **delete `eval`, replace `html` with A2UI message types, and bundle a reference renderer** — and otherwise leave glimpse alone.

## What This Becomes

A **single-binary A2UI client appliance**. Pipe spec-compliant A2UI JSONL to its stdin. A native window appears, renders trusted components from a catalog, and emits user actions back on stdout. Provider-neutral, MCP-pairable, drop-in for any agent runtime that can spawn a subprocess.

## Decisions

### Keep (verbatim from glimpse)

- Per-platform native binary model (Swift on Mac first; Linux/Windows deferred).
- JSONL stdio framing.
- Lifecycle events: `ready`, `closed`.
- Custom URL scheme for asset serving (already there for sandbox / path-traversal protection — perfect for serving the embedded renderer offline).
- Single-file Swift discipline.
- Zero-runtime-dependency philosophy.
- "Compile on install" via swiftc.

### Swap (the actual work)

| Upstream command | Fork replacement |
|------------------|------------------|
| `{type:"html", html:"<base64>"}` | A2UI v0.8 message types (likely `surfaceUpdate`, `dataModelUpdate`, etc. — confirm against `specification/`) |
| `{type:"eval", js:"..."}` | **Removed from public surface.** Optionally retained behind `--unsafe-eval` flag for renderer dev/debug. |
| `{type:"message", data:...}` (inbound) | A2UI `action` messages with structured `componentId` + `payload` |

### Add

- **Bundled A2UI reference renderer.** Vendor the Lit web renderer from `google/A2UI/renderers/`. Embed as a static asset in the binary; serve via the existing custom URL scheme; load on WebView init. The native side then just pipes A2UI messages into `window.a2ui.dispatch()` (or whatever the renderer exposes — confirm).
- **Multi-surface support.** A2UI's native concept. One window, N surfaces (tabbed/stacked). Better than glimpse's current 1-window-per-process model.
- **A2UI v0.8 spec compliance suite.** Borrow from `google/A2UI/eval/` if usable; otherwise hand-roll.

### Subtract

- Raw HTML payload as the public interface (kept internally for renderer bootstrap only).
- `eval` from the public surface.
- Anything that assumes "the agent can ship arbitrary JS."

## Scrappy MVP Path

1. **Mac-only.** Defer Linux/Windows. Justification: cross-platform parity is already lumpy upstream and we don't owe it to anyone yet.
2. **Vendor the Lit renderer.** Don't reimplement. Let Google own spec interpretation.
3. **Wire one surface.** Get `surfaceUpdate` → render → action-back working end-to-end with one component (a Button) before adding anything else.
4. **Wrap in an MCP server.** ~150 lines TS. Holds the binary alive across tool calls. Exposes `a2ui.surface_update` / `a2ui.data_update` tools to Claude Code or any MCP client.
5. **Test from Claude Code.** "Show me a confirm dialog with three options" should round-trip.
6. Only after that works, consider Linux/Windows or v0.9.

Estimated effort to step 5: a weekend, not a month. The fork diff on the Swift side is small — a couple hundred lines.

## Costs (Honest)

- **Three-codebase tax (deferred but real).** Forking means owning Swift + Rust + .NET if/when we want parity. Mac-first dodges this initially.
- **Renderer bundle size.** Lit web renderer is probably 200KB–2MB minified. Acceptable, but it dings glimpse's "tiny binary" pitch slightly. Mitigation: ship as separate asset, not a baked-in resource.
- **A2UI v0.8 is a moving target.** Pin to v0.8 explicitly; revisit when v0.9 ships stable.
- **Solo upstream (HazAT).** If we want to upstream the A2UI mode, file an issue first to gauge interest. If they decline, fork is fine — MIT license, no friction. Either way we own divergence cost.

## Risks

- **Lit renderer assumes a web-app context.** May poke at globals, fetch network resources, or assume specific browser APIs. Validate offline operation under WKWebView's custom URL scheme before committing.
- **Spec ambiguities at v0.8.** Public preview means edge cases will bite. Mitigation: vendor the renderer and let it be the de facto spec.
- **A2UI traction failure.** Could remain Google-flavored and never become broadly used. Mitigation: even as a frozen vendored protocol, this is a useful artifact (declarative beats raw-HTML for any agent).

## Competing Hypotheses

1. **"Just use glimpse as-is and put A2UI in the payload."** Build a thin wrapper that accepts A2UI on its own channel and emits HTML/eval calls underneath. *Why we rejected:* keeps `eval` in the trust path, defeats the security thesis of A2UI. The whole point is no-executable-code-across-the-boundary.
2. **"Skip the fork, build a Tauri/Electron sidecar from scratch."** *Why we rejected:* glimpse already solved cross-platform native shell with no Electron weight. Reinventing it is pure cost.
3. **"Wait for Anthropic / Claude Code to ship a canvas natively."** *Why we rejected:* unbounded timeline; provider lock-in; doesn't solve for non-Claude agents.

## Open Questions to Resolve Before Implementation

- Exact A2UI v0.8 message-type schema — read `google/A2UI/specification/` first thing in the implementation session.
- Lit renderer entry point and dispatch API — read `google/A2UI/renderers/`.
- Does the Lit renderer require network/CDN at runtime, or can it run fully offline from `glimpse-a2ui://` ?
- What does "multi-surface" look like in the renderer — single page with regions, or one renderer instance per surface?

## Naming

`glimpse-a2ui` (repo). Binary: open question — `glimpse` (rename), `a2g`, `a2glimpse`. **Boring obvious wins.** Probably keep `glimpse` as the binary name and let "a2ui" be the mode/repo identity. Decide at implementation time.
