# r2-shadcn-renderer — fork rationale, architecture, alternatives

## TL;DR

The vendored Google Lit-renderer in `src/a2glimpse-host.html` made theming a multi-week archaeology project. Every meaningful aesthetic change required spelunking shadow-DOM scoping pathologies, monkey-patching component class prototypes, or filing upstream and waiting. **The protocol is renderer-agnostic.** We forked the renderer to a Vite-built React app using shadcn/ui primitives and Tailwind v4, preserving the wire format and the Swift host bridge contract intact. Two commits: `c673c76` (renderer fork itself) and `2248726` (rich markdown + diff highlighting + hljs syntax + auto-grow fix).

The fork lives on branch `r2-shadcn-renderer` in worktree `glimpse-a2ui-shadcn`. Not yet merged to `main`.

## Why we forked — confidence: VERIFIED

Three weeks of design-track session work documented exactly how painful Lit-renderer theming is. The receipts:

- **`knowledge/20260509-214700.shadow-dom-css-scoping-pathology.knowledge.md`** — buttons live two shadow roots deep from `<body>`; selectors don't cross. Required injecting `<style>` into `a2ui-surface.shadowRoot` via `firstUpdated` monkey-patch.
- **`knowledge/20260509-220000.investigation-methodology-and-lever-vocabulary.knowledge.md`** — five distinct levers (custom properties on body, `additionalStyles` per component, `@font-face`/global resources, `wrapperPatch`, shadow-scoped `<style>` injection) — each with their own reach limitations and gotchas.
- **`knowledge/20260509-154525.vendored-renderer-pathologies.knowledge.md`** — three confirmed upstream renderer bugs (Tabs theme-gap, MultipleChoice schema/element mismatch, fenced pre block content) we cannot fix without forking the IIFE itself.
- **A whole session on a white-on-white fenced code block** — the bug was a one-line CSS typo in injected shadow CSS, but tracking it down required pixel-sampling, three rounds of cause-isolation (selector match → literal !important → probe-property), and re-injecting CSS into shadow roots. In the shadcn renderer, fenced blocks Just Work because there are no shadow boundaries between the `<pre>` and the body palette.

A2UI v0.8 is **a wire protocol**, not a renderer. JSON describes components (`Button`, `Card`, `Text`, `Row`, `Column`, etc.). Anything that consumes the JSON and emits visible UI satisfies the spec. Google's reference Lit renderer is one option; ours is now another.

## Architecture — confidence: VERIFIED

```
glimpse-a2ui-shadcn/
├── renderer/                       ← Vite project (React 19 + TS + Tailwind v4)
│   ├── src/
│   │   ├── App.tsx                 ← root: subscribes to store, renders visible surfaces
│   │   ├── main.tsx                ← React mount
│   │   ├── index.css               ← @theme palette + global body styles + hljs theme
│   │   ├── a2ui/
│   │   │   ├── schema.ts           ← A2UI v0.8 wire types (subset, only what's consumed)
│   │   │   ├── store.ts            ← in-memory store + pub/sub for useSyncExternalStore
│   │   │   ├── bridge.ts           ← WKWebView bridge (in: dispatch, out: postMessage)
│   │   │   ├── Renderer.tsx        ← component-tree walker → shadcn primitives
│   │   │   ├── Markdown.tsx        ← react-markdown wrapper + diff renderer + hljs hookup
│   │   │   └── highlight.ts        ← lowlight (hljs as hast tree) + curated language pack
│   │   ├── components/ui/          ← shadcn-vendored: button card input slider checkbox
│   │   │                             label tabs dialog radio-group
│   │   └── lib/utils.ts            ← cn() helper
│   ├── components.json             ← shadcn CLI config
│   ├── vite.config.ts              ← singlefile + tailwind plugins, @ → src alias
│   ├── tsconfig.{json,app,node}
│   └── package.json
├── scripts/
│   └── build-renderer.mjs          ← runs `vite build` → copies dist/index.html
│                                     to ../src/a2glimpse-host.html (and .app's
│                                     Resources/ if present)
└── src/
    ├── a2glimpse-host.html         ← bundled output (generated, ~565KB single-file)
    └── a2glimpse.swift             ← UNCHANGED from main
```

**Build pipeline:**

```
npm run build:renderer     # vite build → singlefile bundle → src/a2glimpse-host.html
npm run build:macos        # swiftc → src/a2glimpse
npm run build:app          # build-app.mjs → dist/a2glimpse.app/
```

The Swift binary loads `a2glimpse-host.html` via `webView.loadFileURL`. From Swift's perspective, this fork is invisible — same load path, same bridge surface.

## Bridge contract — confidence: VERIFIED

The WKWebView ↔ Swift bridge is preserved verbatim from the Lit-era renderer. Inbound (Swift → page) and outbound (page → Swift) shapes are unchanged. This is what makes the fork drop-in: the MCP server, the Swift host, every test fixture's JSONL, and the entire wire protocol stay the same.

**Inbound — Swift evals JS that calls:**
```js
window.a2glimpse.dispatch(<json>)   // surfaceUpdate / dataModelUpdate
                                    // / beginRendering / deleteSurface
```

**Outbound — page calls:**
```js
window.webkit.messageHandlers.glimpse.postMessage(JSON.stringify({
  __a2glimpse_host_ready: true     // Swift waits for this before flushing
                                   // queued messages from raw-stdin
}))
window.webkit.messageHandlers.glimpse.postMessage(JSON.stringify({
  userAction: { name, surfaceId, sourceComponentId, timestamp, context }
}))
window.webkit.messageHandlers.glimpse.postMessage(JSON.stringify({
  __a2glimpse_resize_to_content: true, height, width
}))
```

The `__a2glimpse_host_ready` magic message (not a DOM event) is the load-bearing signal. **Don't omit it.** The Lit IIFE emits this from its host element's `connectedCallback`; the shadcn renderer emits it from `installBridge()` in `bridge.ts` after wiring `window.a2glimpse.dispatch`.

## Theming — confidence: VERIFIED

**One file (`src/index.css`) controls the entire palette.** Tailwind v4 `@theme` block defines the workshop ladder (paper / paper-up / card / card-up / ink / ink-2 / line / line-soft / voltage / voltage-2 / spark / danger), aliased to shadcn's expected token names (`--color-background`, `--color-primary`, `--color-card`, `--color-border`, `--color-ring`, etc.). Every shadcn primitive consumes those automatically.

To change the palette: edit one block. No per-component overrides. No shadow-DOM injection. No wrapperPatch.

## Alternatives considered

### Path B — Tailwind paint-over of the Lit renderer (rejected)

Keep the Lit IIFE, inject Tailwind utilities + token vars into the body, target `a2ui-*` host elements via the existing `additionalStyles` + custom-property mechanism. **Rejected** because:

- Inner shadow-DOM components (the actual `<button>`, `<input>`) stay un-styleable from outside without monkey-patching every Lit class.
- Markdown engine stays vendored — no fix for the white-on-white pre block, the empty MC dropdown, the fenced-block content bug.
- Three weeks of design-track session work demonstrated this path's ceiling. Past it, every aesthetic question becomes a forensic exercise.

### Path C — Replace just the markdown engine, keep Lit for components (rejected)

Vendor a custom markdown renderer that injects into `a2ui-text.shadowRoot`, leave everything else Lit. **Rejected** because:

- Theming the rest of the surface is still painful; we'd be solving 10% of the problem.
- The injection mechanism is the same wrapperPatch the rest is.
- If we're going to write a markdown renderer (we did, in the shadcn path), might as well swap the whole thing.

### Path D — Wait for upstream A2UI to fix the renderer (rejected)

Google's `@a2ui/lit` ships on its own cadence. Filed bugs (B1) take indeterminate time. **Rejected** because:

- We can't promise users a polished surface on a someone-else schedule.
- The protocol is the contract; the renderer is implementation. Nothing in the spec says we have to use Google's Lit.
- A renderer fork doesn't preclude tracking upstream — we can re-vendor at any time if we want to.

## Tradeoffs

**Bundle size:** 565KB / 171KB gzipped. The Lit IIFE was 312KB / 88KB gzipped. We pay ~250KB for: shadcn/Tailwind/radix primitives, react-markdown + remark-gfm, lowlight + 16 hljs languages. Reasonable for an agent surface that needs to render code + prose + interactive controls.

**Component fidelity:** shadcn components are battle-tested. radix primitives behind Slider/Tabs/Dialog give us correct keyboard navigation, focus management, ARIA semantics — things the Lit renderer either had bugs in or didn't have at all.

**Maintenance surface:** we now own a renderer. If A2UI v0.8.x ships a new component or schema field, we add it (vs the Lit path, where we'd revendor and re-apply patches). Net: about the same maintenance effort, with full design control as the upside.

**Cross-platform:** shadcn renderer is web-platform code — no macOS-specific assumptions. If a2glimpse ever expands beyond WKWebView (Linux Electron-like, Windows WebView2), the renderer comes along for free.

## What's NOT done — confidence: VERIFIED

- **mcporter symlink** — `/opt/homebrew/lib/node_modules/a2glimpse` is symlinked to the **design** worktree. To drive the shadcn renderer via `mcporter call`, either:
  - Set `A2GLIMPSE_BINARY_PATH=/Users/brahn/src/github.com/bdmorin/glimpse-a2ui-shadcn/src/a2glimpse` for that session, or
  - Re-symlink (`npm install -g .` from the shadcn worktree)
- **Visual goldens** — wholesale invalid against the shadcn renderer. Re-bless required after merge to `main`. Threshold tuning likely needed (anti-aliasing differs, Tailwind reset is different from the Lit reset).
- **Full upstream A2UI component coverage** — schema.ts only types what we consume. Add `Image`, `Audio`, `Video`, `DateTimeInput`, `List`, `Divider` when an agent flow needs them.
- **Action context paths** — `dispatchAction()` resolves `context.value.path` against the surface's data, but only for top-level keys (no `/foo/bar` deep paths yet). Adequate for current flows; revisit when an agent needs nested binding.
- **Multi-surface stack** — App.tsx renders all visible surfaces in order, but the C2 vertical-stack semantics from the Lit era haven't been visually exercised yet.
- **Dev experience** — `npm run dev` starts Vite at localhost:5173 but the bridge throws (no WKWebView). Acceptable; we always test through the Swift host.

## Why this is a fork, not a feature

This is filed under `knowledge/*.analysis.md` because it's a **decision with alternatives weighed**, not a runbook. The fork happens once. Future iteration is conventional React/Tailwind/shadcn work; consult shadcn-ui docs, not this file.

If we ever decide to fold this back to `main` and retire the Lit-era host:

1. Tag the Lit-era host (`v0.8.9-multi-surface-merge` already does this).
2. Merge `r2-shadcn-renderer` to `main`.
3. Re-bless visual goldens at the new host hash.
4. Update HANDOFF.md to reflect that `src/a2glimpse-host.html` is now generated from `renderer/`.
5. Delete `src/a2glimpse-host.html` from the package's `files` array (already auto-generated, but list it explicitly in the build outputs section).
6. Document the build pipeline addition in README.
