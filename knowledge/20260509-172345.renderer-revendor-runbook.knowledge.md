# Renderer Bundle Re-vendor Runbook

> **Status: LIKELY (unproven).** This runbook was authored before any actual
> re-vendor was performed. The first operator who runs it MUST edit it in
> place to capture what actually had to be done. On first successful run,
> bump `confidence` to VERIFIED and add a `verified-by:` frontmatter field.

## 0. Why this exists

`src/a2glimpse-host.html` inlines the entire `@a2ui/lit` renderer as an IIFE
between two `<script>` tags. There is no build step that pulls it; it was
hand-pasted at fork time. When upstream ships a v0.8.x bugfix that fixes
something we currently work around (see `vendored-renderer-pathologies`),
we want to adopt it — but the renderer is the largest piece of third-party
code we ship, and it runs in the WKWebView next to the agent feed. Every
re-vendor is a trust event.

This document makes the re-vendor process auditable and reversible.

## 1. Pinning policy

- We pin to the **v0.8.x** line of `@a2ui/lit`. Upstream's `package.json`
  on `main` has already moved to `0.9.x`; we do **not** track that.
- v0.8 is closed (no longer evolving) but receives bugfixes. Adopt those.
- v0.9 / v0.10 are spec major pivots. **Out of scope.** A migration
  across spec major requires a new plan, not this runbook.
- The authoritative upstream tag for our line is `v0.8` in
  `github.com/google/A2UI`. As of 2026-05-09 that tag points at commit
  `19919ef4c8ad3185867f70386fa4669284d7714c`. Future bugfixes on the v0.8
  branch will move that tag (or land on a `release-0.8` branch); confirm
  on each re-vendor.

## 2. Where the upstream bundle lives

- Repo: `github.com/google/A2UI`
- Subtree: `renderers/lit/`
- v0.8 source: `renderers/lit/src/0.8/` (the v0.9 source lives at
  `renderers/lit/src/v0_9/` and we ignore it)
- Build: `wireit`-driven. `npm run build` from `renderers/lit/` produces
  `dist/`. The bundle output path inside `dist/` was not verified at
  authorship time — **first operator: capture the exact path here.**
- The vendored bytes we have today were authored against v0.8.x sometime
  before the fork. We did **not** record which upstream sha they came from.
  Treat the current bundle as `<unknown upstream sha>` until first
  re-vendor establishes a known good baseline.

## 3. Layout in our repo

`src/a2glimpse-host.html` is one self-contained file. Relevant boundaries
(line numbers as of bundle hash `0ff4d09524c9` — they will shift after
re-vendor; locate by the comments, not the numbers):

| Region | Lines (current) | Owner |
|---|---|---|
| Host theme `<style>` block | 1–194 | OURS — wrapper-level styles |
| `BEGIN/END a2glimpse-markdown-engine` (importmap + base64-inlined `@a2ui/markdown-it` shim) | 196–204 | OURS — auto-generated by `scripts/inline-markdown-engine.mjs`, source `src/markdown/a2glimpse-markdown.mjs`. **Do not hand-edit.** |
| `<a2glimpse-app></a2glimpse-app>` | 207 | OURS |
| Vendored Lit IIFE `<script>...</script>` | 208–11077 | UPSTREAM — replace wholesale on re-vendor |
| `a2glimpse-host-ready-shim` `<script>` | 11078–11118 | OURS — Swift bridge, **preserve verbatim** |
| Tabs theme-gap prototype patch `<script>` | 11119–11144 | OURS — workaround, **review for removal** on each re-vendor |
| Modal auto-open `<script>` | 11145–11183 | OURS — test-mode workaround, **review for removal** on each re-vendor |
| `</body></html>` | 11184–11185 | OURS |

The IIFE is the **only** region replaced wholesale. Everything else is
ours and outlives the re-vendor.

## 4. Re-vendor procedure

### 4.1 Pre-flight

```bash
# From the project root.
git checkout -b revendor-lit-<upstream-sha-short>
git status --short  # must be clean
node test/visual.mjs --only welcome  # confirm current goldens pass before we start
```

If the visual harness already fails on `main`, **stop**. Fix that first;
do not re-vendor on top of red.

### 4.2 Pull and build upstream

```bash
# Adjust UPSTREAM_DIR to wherever your local clone lives.
UPSTREAM_DIR=~/src/github.com/google/A2UI
git -C "$UPSTREAM_DIR" fetch --tags
git -C "$UPSTREAM_DIR" checkout v0.8         # or whichever bugfix tag/sha
UPSTREAM_SHA=$(git -C "$UPSTREAM_DIR" rev-parse HEAD)
echo "Upstream sha: $UPSTREAM_SHA"

cd "$UPSTREAM_DIR/renderers/lit"
npm install
npm run build
# First operator: capture the exact path of the IIFE bundle output here.
# Suspected: dist/index.iife.js or dist/src/index.js bundled via the same
# wireit pipeline. VERIFY before pasting.
```

### 4.3 Trust-boundary diff check

Before pasting anything in, diff the **new** bundle against the **old**
IIFE region for net-new appearances of dangerous primitives. The current
IIFE is bytes 208–11077 of `src/a2glimpse-host.html`.

```bash
# Extract current IIFE for comparison.
sed -n '209,11077p' src/a2glimpse-host.html > /tmp/iife-old.js
# Place the new upstream bundle output at /tmp/iife-new.js.
# Then grep both for the trust-boundary tripwires:
PATTERNS=(
  'eval\('
  'Function\s*\('       # constructor-style dynamic compile
  'unsafeHTML'
  'innerHTML *='
  '\bfetch\('
  'import\('             # dynamic import
  'document\.write'
  'WebAssembly'
  'Worker\('
  'postMessage'
)
for pat in "${PATTERNS[@]}"; do
  echo "=== $pat ==="
  echo "old:"; grep -cE "$pat" /tmp/iife-old.js || true
  echo "new:"; grep -cE "$pat" /tmp/iife-new.js || true
done
```

**Decision rule:** if any of these counts increased OR a new primitive
(not in the list) appears that performs runtime code loading, dynamic
HTML injection, or network I/O — **STOP and escalate to Brian.** Do not
silently adopt. Re-vendoring is a trust event; new powers in the bundle
require explicit human review.

Known acceptable baseline: `unsafeHTML` is already used by the renderer's
`MarkdownDirective` (this is why we vendor a hardened markdown engine via
the importmap shim — see `a2glimpse-markdown-engine` block).

### 4.4 Swap in the new bundle

1. Open `src/a2glimpse-host.html`.
2. Delete lines from immediately after the opening `<script>` on line 208
   through the line just before the closing `</script>` at line 11077.
   Locate by these landmarks, not numbers:
   - **First line to delete:** the `(() => {` immediately after `<script>`.
   - **Last line to delete:** the closing `})();` and license-banner
     comments just before `</script>` on what is currently line 11077.
3. Paste the new IIFE between those tags. Preserve the wrapping
   `<script>` and `</script>` themselves.
4. **Do not touch** the wrapper scripts below the IIFE
   (`a2glimpse-host-ready-shim`, Tabs patch, Modal patch). They live in
   their own `<script>` blocks for a reason.

### 4.5 Compute and record the new bundle hash

```bash
# Whole-file hash (this is what the visual harness keys snapshots on).
shasum -a 256 src/a2glimpse-host.html | awk '{print $1}'
# IIFE-only hash (more durable across wrapper edits — record both).
sed -n '<begin>,<end>p' src/a2glimpse-host.html | shasum -a 256 | awk '{print $1}'
# (<begin>,<end> = the line numbers spanning the new IIFE; the visual
# harness uses createHash on the whole file, but recording the IIFE-only
# hash lets us correlate re-vendors across wrapper edits.)
```

Record both in `knowledge/renderer-bundle-pins.md`. Add a new row with
upstream sha, vendored date, vendored-by, both hashes, and a one-line
summary of the bugfix being adopted.

### 4.6 Workaround re-verification

We carry three host-side workarounds for confirmed renderer bugs (see
`20260509-154525.vendored-renderer-pathologies.knowledge.md`). After
re-vendor, **walk through each one** and decide whether to remove it.
Code we don't carry is code we don't maintain.

| Workaround | Where | Bug it works around | How to verify the fix |
|---|---|---|---|
| Tabs theme-gap prototype patch | `<script>` after host-ready shim, ~lines 11119–11144 | `defaultTheme.components.Tabs` missing `element` and proper `controls.{all,selected}` shape; renderer dereferences and throws inside `Tabs.render()` | Remove the patch, run `node test/visual.mjs --only tabs`. If Tabs renders content (not blank shadow root), the upstream fixed it. Delete the patch and the comment. |
| Modal auto-open (test-mode only) | `<script>` after Tabs patch, ~lines 11145–11183 | Modal renders closed-state `<section slot=entry>` and only mounts `<dialog>` after click; visual harness needs the open state | This is **not** a bug, it's a test affordance. The renderer behavior here is correct. Keep the patch unless we find a non-DOM-click way to open Modal in test mode (unlikely without a new public stdin command, which would punch a trust boundary hole). |
| MultipleChoice fixture-level workaround | In the relevant fixture under `test/fixtures/` (not in host.html) | Renderer's MultipleChoice schema/element mismatch — see pathologies doc for current shape | After re-vendor, try the un-worked-around fixture shape from upstream's spec docs. If it renders, remove the fixture-level workaround. |

For each workaround that survives the re-vendor, leave a comment in the
patch noting the upstream sha you verified against, e.g.
`// Still required as of upstream <sha-short>; <upstream-bug-link>`.

### 4.7 Visual harness re-bless protocol

```bash
node test/visual.mjs
```

Expect **everything to fail.** The harness keys snapshots on the SHA-256
of the entire `src/a2glimpse-host.html` (truncated to 12 hex chars), and
the snapshot dir is `test/__snapshots__/<hash>/`. Replacing the IIFE
guarantees a new hash → no goldens exist for the new dir → all fixtures
report "no golden".

**Do not run `--update` first.** Bless deliberately:

1. Run without `--update` so the harness creates `<name>.actual.png` for
   each fixture but logs missing goldens.
2. **Eyeball every fixture's actual PNG.** For each one:
   - Compare against the previous-hash golden by hand
     (`open test/__snapshots__/<old-hash>/<name>.png` next to
     `<new-hash>/<name>.actual.png`).
   - Decide: is the visual change expected? If yes, write a one-line
     rationale in the dev log entry. If no — investigate, do not bless.
3. Once every fixture has been eyeballed and rationalized, run with
   `--update` to write the new goldens.
4. Commit the new goldens in their own commit on the re-vendor branch
   with message `chore(test): re-bless visual goldens for renderer
   <upstream-sha-short>` so the bless is reviewable separately.

**Bless on the re-vendor branch, never on `main`.** Main's goldens stay
valid for the previous hash until merge.

### 4.8 Run the rest of the suite

```bash
node test/test.mjs   # markdown-engine + anything else under test/
```

If anything fails that isn't a visual test, the IIFE swap broke a
host-side contract — check that you preserved the importmap and the
host-ready-shim, and that the new IIFE still defines the custom elements
the wrapper expects (`a2glimpse-app`, `a2ui-tabs`, `a2ui-modal`).

### 4.9 Commit and merge

Three-commit shape on the re-vendor branch:

1. `chore(vendor): re-vendor @a2ui/lit to <upstream-sha-short>` — the
   IIFE swap and the bundle-pins ledger entry, nothing else.
2. `chore(test): re-bless visual goldens for renderer <upstream-sha-short>`
   — only the new `test/__snapshots__/<hash>/` files.
3. (Optional) `refactor(host): drop <workaround-name>; fixed in
   upstream <upstream-sha-short>` — if any workaround came out.

Open the PR; link to the upstream changelog or the bugfix commit; link
to the bundle-pins entry.

## 5. Rollback

If the re-vendor breaks something we can't fix in-branch:

1. Tag the broken state for forensic value:
   `git tag revendor-failed-<upstream-sha-short>` then push the tag.
2. Stash the upstream IIFE bytes as a knowledge artifact at
   `knowledge/<timestamp>.failed-revendor-<upstream-sha-short>.binary.iife.js`
   (filename carries metadata since this is binary-ish content) so future
   sessions can diff without re-running the upstream build.
3. Delete the re-vendor branch's IIFE-swap commit
   (`git revert <sha>` on the branch, do **not** force-push main).
4. Add a row to `knowledge/renderer-bundle-pins.md` with status
   `ATTEMPTED — REVERTED` and a one-line cause.
5. Append a DEV_LOG entry describing what broke and the symptom — future
   re-vendor attempts will start by reading this.

## 6. First-time gotchas to capture

When this runbook is first executed end-to-end, **edit it in place** to
capture what was actually different from the assessment above. The
likely places it's wrong:

- The exact build output path and bundle filename in `dist/`.
- Whether `npm run build` Just Works on a fresh clone, or requires a
  parent `npm install` at the workspace root first.
- Whether the upstream bundle is already an IIFE or whether we have to
  wrap it ourselves (the current bundle was hand-pasted; upstream may
  ship ESM).
- Whether the `@a2ui/markdown-it` importmap shim still satisfies the
  new bundle's dynamic-import contract (the current shim is documented
  in the `a2glimpse-markdown-engine` block; if upstream changes the
  contract, our shim breaks silently).
- Whether the wrapper-side prototype patches survive name-mangling in
  the new build (`a2ui-tabs` and `a2ui-modal` element names are stable
  across builds; class identities aren't, but we patch via
  `customElements.whenDefined(<tag>)` which is name-stable, so this
  should be fine).
- Whether the new bundle still uses `unsafeHTML` for the Text usageHint
  pipeline. If it stops, our markdown-engine shim becomes a vestigial
  component we should consider removing.

After the first successful run, change frontmatter `confidence: LIKELY`
to `confidence: VERIFIED` and add `verified-by: <agent-or-brian>`,
`verified-date: YYYY-MM-DD`, `verified-against-upstream: <sha>`.
